# 微信公眾號文章：作者-編輯-讀者三Agent協作

對 $ARGUMENTS 進行深度研究，產出一篇可直接釋出的微信公眾號文章。三個Agent各司其職：作者寫深度初稿，編輯精修結構和表達，讀者從目標受眾視角審讀。

**支援輸入格式**：主題描述，例如：`大模型OPD技術解讀`、`Qwen3技術報告解讀`、`為什麼巴菲特不買科技股`

---

## 設計理念

一篇好的公眾號文章需要同時滿足三個維度：
1. **深度**——對得起花時間讀完的人（作者負責）
2. **可讀性**——結構清晰、節奏好、不勸退（編輯負責）
3. **真的能看懂**——目標讀者不會在中途放棄（讀者負責）

單人寫作容易"自嗨"——寫的人覺得清楚，讀的人看不懂。三Agent協作的本質是**強制引入外部視角**。

## 階段一：研究與素材收集

### 第一步：明確文章定位

在開始寫作之前，先確認以下資訊（如使用者未指定則主動詢問）：

| 維度 | 需要確認 | 預設值 |
|------|---------|--------|
| **目標讀者** | 技術背景程度 | 有點技術背景但非該領域專家 |
| **文章深度** | 科普/中深度/硬核 | 中深度（有公式但要解釋清楚） |
| **文章長度** | 字數範圍 | 3000-4000字 |
| **是否需要下載原始論文/資料** | 需要PDF/配圖 | 是 |
| **寫作風格** | 正式/對話式/犀利 | 對話式（像寫給聰明的朋友） |

### 第二步：深度研究

使用 Agent 工具**並行**啟動2-3個研究Agent，收集足夠的素材：

**研究Agent A：核心內容研究**
- 如果是論文解讀：下載論文PDF、提取核心貢獻、關鍵圖表、實驗結果
- 如果是技術主題：搜尋最新進展、關鍵論文、技術細節
- 如果是商業/投資主題：搜尋最新資料、行業報告、競爭格局

**研究Agent B：行業背景與應用**
- 搜尋該技術/主題的行業落地情況
- 哪些公司在用？效果如何？
- 最新的發展趨勢和里程碑事件

**研究Agent C（可選）：競品/對比研究**
- 同類方法/產品的對比
- 歷史發展脈絡
- 未來演進方向

### 第三步：整理素材框架

研究Agent全部完成後，整理出：
1. **核心論點**（一句話概括文章要傳達的核心資訊）
2. **關鍵資料**（3-5個最有衝擊力的資料點）
3. **配圖清單**（需要哪些圖，來源是什麼）
4. **文章大綱**（6-8個章節的標題和核心內容）

## 階段二：作者Agent寫初稿

使用 Agent 工具啟動**作者Agent**，給出詳細的寫作指令。

### 作者Agent的Prompt模板

```
你是一位深度技術寫作者（Author Agent），需要寫一篇微信公眾號文章。

## 目標讀者
{根據第一步確認的讀者畫像}

## 寫作風格要求
- 純中文表達，避免中英文夾雜（技術術語首次出現時給英文，後續用中文）
- 像寫給聰明的朋友看的技術科普，不是學術論文翻譯
- 用類比幫助理解，但類比要貼切、不俗套
- 關鍵公式/資料要有，但每個都要用大白話解釋
- 不用emoji
- 段落不超過4行（公眾號閱讀環境）

## 核心內容
{整理好的素材、資料、論點}

## 文章結構要求
1. **開頭（前3段）**：必須有強鉤子——用資料衝擊力或反直覺的結論開場，不要用溫和的類比開場
2. **背景**：為什麼這件事重要？解決什麼問題？
3. **核心內容（2-3節）**：技術深度在這裡體現，但每個技術點都要有"大白話翻譯"
4. **實證/案例**：用資料和案例說話，不要空談
5. **行業影響/展望**：這件事對行業意味著什麼
6. **結尾**：一句有傳播力的判斷收束，適合被截圖轉發

## 配圖要求
- 論文解讀類文章：必須從論文PDF中提取原圖，直接用 `![描述](相對路徑)` 插入文章，不要用 [圖X：描述] 佔位符
- 提取方法：用 pdftoppm 將PDF頁面渲染為高解析度PNG（至少900 DPI），再用 PIL 裁切目標圖表區域
- 每張圖不小於500KB，確保高畫質
- 圖片統一存放在 `assets/{主題簡稱}/` 目錄下
- 非論文類文章：如需配圖，搜尋並下載合適的圖片，同樣直接插入

## 公式要求
- 所有數學公式使用 LaTeX 格式：行內用 `$...$`，獨立公式用 `$$...$$`
- 禁止用純文字寫公式（如 `> D_KL(P || Q) = ...`），必須用 LaTeX 渲染格式
- 每個公式後面仍然要配"大白話翻譯"

請寫出完整的文章初稿，約{目標字數}字。
```

### 作者Agent完成後

檢查初稿檔案是否生成，閱讀全文確認內容完整性。

## 階段三：編輯Agent + 讀者Agent並行審閱

初稿完成後，使用 Agent 工具在**同一條訊息**中啟動編輯Agent和讀者Agent。

### 編輯Agent的Prompt模板

```
你是一位資深公眾號編輯（Editor Agent）。請對以下文章進行精修審閱。

## 審閱標準
1. **標題**：是否在朋友圈能吸引點選？是否會被截斷（超過30字）？
2. **開頭**：前3段能否留住讀者？鉤子是否足夠強？
3. **結構**：邏輯鏈是否流暢？有無跳躍或斷層？
4. **深度與可讀性平衡**：公式/技術部分是否真的通俗？有無"假裝通俗但沒解釋清楚"的地方？
5. **節奏**：有無太長的段落？每節長度是否合適？
6. **配圖**：圖片是否已實際插入（非佔位符）？位置是否在讀者最需要視覺輔助時出現？
7. **結尾**：有無傳播力？讀者看完會想轉發嗎？

## 文章全文
{完整初稿}

## 輸出格式
1. 總體評價（3-5句話）
2. 標題修改建議（給2-3個備選）
3. 逐節修改建議（給出"原文→建議修改為"的具體對照）
4. 最關鍵的3個改進點
```

## 階段四：定稿

### 第一步：綜合兩個Agent的反饋

重點關注以下高頻問題：

| 問題型別 | 編輯常見反饋 | 讀者常見反饋 | 處理方式 |
|---------|------------|------------|---------|
| 開頭太弱 | 鉤子不夠強 | 前3段沒動力繼續 | 用資料/反直覺結論重寫開頭 |
| 技術段勸退 | 公式太密集 | 某段需要重讀3遍 | 刪公式或圖片化，加更直覺的類比 |
| 節奏拖沓 | 某節太長 | 某處失去耐心 | 合併或刪減（尤其是後半段重複的技術解釋） |
| 結尾無力 | 缺傳播力 | 不會轉發 | 重寫為一句可截圖傳播的判斷 |
| 概念跳躍 | 邏輯有斷層 | 某處"突然看不懂了" | 補過渡句或背景解釋 |

### 第二步：執行修改

根據反饋重寫文章。核心修改原則：

1. **編輯和讀者都指出的問題，必須改**
2. **只有編輯指出的問題，大機率要改**（編輯的專業判斷通常準確）
3. **只有讀者指出的問題，視情況改**（讀者反饋代表真實體驗，但不一定每條都需要響應）
4. **兩者矛盾時，偏向讀者**（編輯追求完美，但讀者體驗是最終標準）

### 第三步：提取配圖

論文解讀類文章必須在定稿前完成配圖提取：

1. **渲染**：`pdftoppm -png -r 900 -f {頁碼} -l {頁碼} 論文.pdf /tmp/page`（900 DPI起步，如果圖片不到500KB則升到1200或1500 DPI）
2. **定位**：先用150 DPI渲染全頁，目視確認各圖表的畫素座標
3. **裁切**：用 PIL 按座標裁切，`compress_level=1` 儲存，確保每張 ≥ 500KB
4. **儲存**：儲存到 `assets/{主題簡稱}/` 目錄，命名為 `fig{序號}-{描述}.png`
5. **插入**：文章中用 `![描述](../../assets/{主題簡稱}/fig{序號}-{描述}.png)` 引用

### 第四步：產出最終檔案

將定稿儲存為 md 檔案，檔案末尾附上論文/資料原文連結：

```markdown
**論文原文：**
- arXiv: {連結}
```

## 寫作紅線

1. **不虛構資料**。引用的資料必須有來源，搜不到就標註"估計"
2. **不用AI腔調**。禁止"讓我們一起來看看"、"值得關注的是"、"不得不說"等套話
3. **不過度承諾**。技術文章不說"顛覆性的"、"革命性的"，用資料說話
4. **公式必須配大白話**。每個公式後面都要有一段"翻譯成人話就是……"
5. **公式必須用LaTeX**。`$$...$$` 格式，禁止純文字公式
6. **配圖必須實際插入**。論文解讀類從PDF提取高畫質原圖（≥500KB），禁止用 `[圖X]` 佔位符
7. **表格括號註釋要精確**。描述概念時用準確的定義，不用模糊的動賓短語（如"文字來自教師"而非"學教師的文字"）
8. **類比要一以貫之**。全文用一個主線類比，不要每節換一個新類比
9. **結尾必須有傳播力**。最後一句話要值得被單獨截圖轉發
