## 0. 摘要

**Mid-train 不是 post-train 的放大版，也不是簡單把 pretrain 再跑一段。** 本文采用一個保守的工程定義：

> Mid-train 是 base pretrain 後、SFT/RL 等 post-train 前的一段受控繼續訓練，用來改變模型的能力分佈、上下文適應能力，以及後續 SFT/RL 可利用的可訓練性。

公開資料中，OLMo 2 明確把 base pretraining 之後、post-training 之前的階段稱為 mid-training；Qwen3、DeepSeek-V3、Qwen2.5-1M 等報告並不總使用這個詞，而是稱為 pre-training、continued pre-training、context extension 或 long-context pre-training。本文只在工程意義上把這些階段歸入 mid-train-like 範疇，不能等同於原報告術語。

資料選擇上，本文不做模型歷史覆盤。每個模型家族優先讀最新一版公開技術報告；只有當上一版披露了當前版本沒有替代的資訊，例如 1M context、顯式 mid-training 命名、annealing 或 synthetic data 細節時，才保留上一版或必要例外。執行專案時，應先用目標模型家族的最新報告替換本文中的歷史樣本。

實踐上，新團隊不應先追大規模訓練，而應先做小模型 ablation、資料桶驗證、汙染檢查、base 迴歸評估，再用主模型短程訓練驗證方向，並用固定 post-train recipe 檢驗收益是否保留。**Mid-train 的成功標準不是某個 base benchmark 上漲，而是能力增強、遺忘可控、post-train 後收益仍存在。**

邊界上，本文不把所有 continued pretraining 都等同於 mid-train，也不把 DCA、YaRN、sparse attention、chunked prefill 等推理或架構技巧簡單歸為訓練配方。公開案例只能作為設計參考，不能照搬為 recipe。

文中的資料比例、token budget、90 天路線圖、質量門和事故 runbook 均為面向新團隊的工程起步建議，不是公開論文直接給出的通用結論。

對一個有 post-train 經驗、準備承擔 mid-train 的基模團隊，**最重要的不是立刻上大規模訓練，而是先建立四件事**：

1. **明確目標能力和不可犧牲能力**：例如數學提升不能以通用對話、事實知識、多語能力大幅退化為代價。
2. **做小模型和短程 ablation**：先驗證資料桶、學習率、序列長度、汙染控制和遺忘風險。
3. **建立 mid-train 專用評估**：不能只看 pretrain loss，也不能只看 post-train 後的榜單。
4. **讓 Post-train DRI 接力驗證**：mid-train 的成功標準之一，是 SFT/RL 後收益仍能保留、訓練更可控，而不只是 base checkpoint 某幾個 zero-shot 分數更好。

### 0.1 讀者決策樹：你到底該做哪類 mid-train

| 目標 | 優先路線 | 第一輪實驗 | 關鍵評估 | 不建議一開始做 |
| --- | --- | --- | --- | --- |
| 數學、程式碼、STEM、複雜 QA | Capability mid-train | 小模型資料 mix + 低 LR 短程訓練 | 目標能力提升 + 通用能力回撤 | 同時拉長到 128K/1M |
| 32K/128K 長上下文 | Context extension | 逐步長度擴充套件 + 長文件資料桶 | 多證據、長文件、長程式碼任務 | 只看 needle-in-a-haystack |
| RL reasoning 上限 | Reasoning substrate mid-train | 驗證 base 是否已有 edge-of-competence | 固定 RL/RLVR recipe 接力對比 | 把 RL 當成能力來源本身 |
| 垂直領域能力 | Domain continued pretraining | 低比例領域資料 + 通用保真資料 | 領域任務 + 通用迴歸 | 用單一領域語料灌滿訓練 |
| Coding agent / tool use | Code/FIM/trajectory mid-train | FIM、repo-level completion、多檔案定位 | repo task、編輯任務、post-train 後 agent eval | 把互動格式全部前移到 mid-train |

這個決策樹是執行入口，不是排他分類。一個成熟專案最後可能同時包含 capability shaping、long-context adaptation 和 post-train readiness，但**第一輪最好只驗證一個主假設**。

### 0.2 第一天怎麼啟動

如果今天就要啟動 mid-train 專案，**不要先開訓練。第一天只做 7 件事**：

1. 拉齊 Model、Data、Training Infra、Eval、Post-train、Safety/Compliance、Product/Domain 的 DRI。
2. 寫一頁 Mid-train charter：目標能力、不可犧牲能力、為什麼必須在 mid-train 做、預算邊界。
3. 凍結 eval registry 的 v0 owner 和 schema；訓練前再凍結 v1 eval registry，包括 base eval、專項 eval、長上下文 eval、私有 holdout、人工樣例審查。沒有 v1，不進入 Gate 1。
4. 讓 Post-train DRI 凍結 paired protocol 的 v0 owner 和 schema；訓練前再凍結 v1 paired protocol，包括 SFT/RL 資料版本、訓練預算、seed、harness、評估集和失敗判定。
5. 建立實驗台賬 schema，規定沒有台賬的實驗不進入結論。
6. 定義 Gate 0-5 的透過條件、阻斷條件和簽字人。
7. 明確第一週不允許做什麼：不邊看私有 holdout 邊調參，不在同一個實驗 ID 下改 mix/過濾器/eval 版本，不把 proxy 成功當成主模型必然成功。

## 1. 術語邊界：什麼是 mid-train

### 1.1 一個實用定義

本文采用工程定義，而不是術語潔癖式定義：

> Mid-train 是 base pretrain 後、post-train 前的繼續訓練階段。它仍主要使用語言建模目標或其變體，但資料、學習率、序列長度和能力目標已經從“最大覆蓋世界文字”轉向“高質量能力塑形”。

這一定義覆蓋幾類公開實踐：

- late-stage pretraining / annealing：在較低學習率下，用高質量資料繼續訓練，追求更低損失和更強下游能力。
- domain / capability continued pretraining：圍繞程式碼、數學、STEM、醫學、法律、金融、多語等能力繼續訓練。
- long-context extension：在原 base context 上擴充套件到 32K、128K、256K 等，配套位置編碼、attention 最佳化和長文件資料；若目標是 1M 級上下文，應單獨驗證訓練長度、位置/attention 外推、chunked prefill、KV cache 和 serving 成本，不能寫成單純繼續訓練或資料配方。
- reasoning-oriented mid-train：用數學、程式碼、合成推理、教材式資料、QA/CoT 等讓 base model 更適合後續 RL 或 RLVR。

Mid-train 和 continued pretraining 有重疊，但強調點不同：continued pretraining 描述“訓練從哪裡接著跑”；mid-train 更強調“為什麼跑、用什麼資料塑形、如何服務 post-train”。

### 1.2 它和 post-train 的分工

Post-train 團隊容易把 mid-train 想成“更多 SFT 資料”或“更早一點的 SFT”。這是危險的，因為兩者最佳化物件不同。

Mid-train 關注：

- 模型內部是否有解決問題所需的知識、演算法模式和表徵能力。
- 長上下文中位置編碼、attention、檢索和跨段歸納是否可用。
- 程式碼、數學、工具軌跡、複雜指令的基礎分佈是否進入模型引數。
- 後續 SFT/RL 是否更容易把能力激發出來。

Post-train 關注：

- 指令遵循、回答風格、安全拒答、偏好排序。
- 多輪對話和使用者意圖對齊。
- 推理過程是否外顯、是否符合產品策略。
- RL/RLHF/RLAIF/RLVR 中的 reward 可最佳化行為。

粗略說，mid-train 更偏向改變能力可達性和引數內分佈，post-train 更偏向改變任務介面、行為策略和產品呈現；但在 reasoning、tool use、agent 和長上下文任務上，兩者會交疊，必須用 paired protocol 判斷收益來自哪裡。

### 1.3 不宜把所有訓練都叫 mid-train

本文不把以下情況簡單歸為 mid-train：

- 從零開始的大規模通用預訓練。
- 單純為了對齊、安全或格式的 SFT。
- 只在 post-train 資料上繼續跑語言建模，但沒有能力目標、資料控制和迴歸評估。
- RAG、外部工具、推理時 search 等不改變模型引數的系統工程。

邊界要清楚，否則團隊會用 post-train 的評估和組織方式管理 mid-train，最後很難判斷收益來自哪裡。

### 1.4 給有 post-train 經驗的基模團隊的 10 條差異認知

1. **Mid-train 的物件是能力分佈，不是回答風格。**
2. Mid-train 的主損失通常仍接近語言建模目標，不是偏好最佳化。
3. Mid-train 資料可以包含 QA/CoT/FIM，但**不要把 SFT 資料整體前移**。
4. **Mid-train 的好 checkpoint 不一定是 loss 最低 checkpoint**，而是能力提升和遺忘之間的 Pareto 點。
5. **Mid-train 的收益必須在 base eval 和 post-train readiness 中同時觀察。**
6. **資料汙染在 mid-train 中比 SFT 更危險**，因為它會進入 base 能力判斷。
7. 合成資料要先被驗證和分層，不應因為”便宜”而提高比例。
8. 長上下文能力要區分訓練長度、推理外推長度和線上可服務長度。
9. **RL 的收益通常依賴 base 已有可達能力區間**，不能把所有 reasoning 問題推給 RL。
10. **放大訓練前必須有停止條件**；mid-train 不是”預算還沒花完就繼續跑”。

## 2. 公開證據強弱分層

公開材料不完全一致，最容易出錯的地方是把“報告明確寫出的訓練階段”“研究者的歸納”和“訪談裡的行業經驗”混為一談。本文按證據強弱和模型版本新舊兩條線使用資料：同一模型家族優先最新一版，最多上一版；更早材料只在有不可替代證據時作為例外。

### 2.0 本文的證據標籤

為避免把公開事實和工程推斷混在一起，後文預設使用四類證據標籤：

| 標籤 | 含義 | 能用於什麼 | 不能用於什麼 |
| --- | --- | --- | --- |
| 報告事實 | 技術報告、論文或官方材料直接披露的階段、token、context、資料型別、訓練目標 | 描述某團隊實際做了什麼 | 推出其它團隊應照搬同一配方 |
| 論文歸納 | 綜述或受控實驗給出的機制、taxonomy、變數關係 | 設計內部 ablation、建立假設 | 直接升級為產業級訓練定律 |
| 工程歸納 | 本文基於多個案例抽出的保守實踐建議 | 制定專案流程、gate、台賬和風險控制 | 聲稱是某篇報告原文結論 |
| 待驗證假設 | 訪談、部落格、行業口徑或小樣本實驗啟發的方向 | 進入小模型/短程實驗候選 | 進入主模型放大決策 |

讀者在使用本文時，應把“報告事實”當作事實底座，把“論文歸納”和“工程歸納”當作實驗設計輸入，把“待驗證假設”當作 backlog。任何會消耗大規模訓練預算的決策，都應回到團隊自己的 data manifest、eval registry 和 post-train paired protocol。

### 2.1 強證據：技術報告明確披露的階段設計

強證據指模型團隊在技術報告中直接披露的訓練階段、token 規模、上下文長度或資料策略。

| 案例 | 使用級別 | 報告原術語 | 公開報告直接支援 | 本文工程類比 |
| --- | --- | --- | --- | --- |
| Qwen3 | 主讀 | pre-training stages | General / Reasoning / Long Context 三個 pre-training stage；約 36T 總量、約 5T reasoning-stage tokens、32K 長上下文訓練 | mid-train-like capability shaping + context adaptation |
| DeepSeek-V3 | 主讀；有同家族新報告時替換 | long context extension | 14.8T base pre-training 後，使用 YaRN 做兩段各 1000 steps 的 4K -> 32K -> 128K context extension | mid-train-like long-context extension，但主要支援長度階段化結論 |
| GLM-4.5 | 主讀 | mid-training | 23T pre-training 後約 1.1T mid-training tokens，顯式分三個子階段：repo-level code（~500B，4K→32K）、synthetic reasoning（~500B）、long-context + agent（~100B，→128K）；MoE 355B/32B | 顯式 mid-training 命名 + 多子階段能力塑形 + 漸進式 context extension；MoE 架構下 mid-train 的參考樣本 |
| Qwen2.5-1M | 上一版專項保留 | long-context pre-training / post-training / inference scaling | progressive long-context pre-training、長上下文 post-training、DCA/YaRN 外推、稀疏注意力和 chunked prefill 部署最佳化 | context adaptation + inference/deployment 協同樣本 |
| OLMo 2 | 必要例外 | mid-training | pretraining -> mid-training -> instruction/preference tuning 的清晰分層；Dolmino Mix、50B/100B/300B、micro-annealing 等細節較透明 | 狹義 mid-training 的公開參考樣本 |
| Phi-4 | 必要例外 | pretraining / midtraining / post-training | 合成資料貫穿 pretraining/midtraining/post-training；midtraining 明確把 4K context 擴到 16K | synthetic-heavy training strategy + context extension 邊界樣本 |
| Llama 3 | 降級參考 | initial pre-training / long-context pre-training / annealing | 405B 在 15.6T tokens 上 initial pre-training，約 800B tokens long-context pre-training 到 128K，最後 40M tokens annealing | late-stage continued pre-training 和 annealing 的保守參考 |

這些報告能支撐“階段設計”的存在，但通常不能支撐“照搬資料比例或學習率”。未公開的 tokenizer、資料清洗、取樣策略、基礎 checkpoint 狀態、最佳化器狀態和系統實現，都會顯著影響結果。

### 2.2 中證據：綜述和受控實驗的歸納

Mid-training survey、PRISM 以及 pre-training / mid-training / RL interplay 這類研究能幫助團隊理解變數關係：資料保留與能力遷移、能力是否位於模型可達邊界、RL 是否能從 mid-trained checkpoint 獲益等。

這些研究的價值是提出機制假設和實驗維度，而不是給出可直接外推的產業配方。尤其是受控 synthetic task 或小規模模型上的結論，不能直接升級成“所有大模型訓練定律”。

### 2.3 弱證據：訪談、部落格和行業口徑

訪談和部落格通常能提供兩類資訊：術語如何被從業者使用，以及團隊如何描述訓練階段的組織邏輯。但這類資料常常省略失敗實驗、資料細節和工程約束，所以只能作為趨勢背景，不能作為硬事實。

如果訪談聲稱某個階段“非常重要”，本文會把它當作待驗證假設；只有技術報告、可復現實驗或團隊內部 ablation 支援後，才進入訓練決策。

### 2.4 從公開資料抽出的保守共識

第一，mid-train-like 階段通常發生在 base 能力形成之後、post-training 之前。Qwen3、DeepSeek-V3、GLM-4.5 是主讀樣本；Qwen2.5-1M、OLMo 2、Phi-4、Llama 3 分別作為上一版專項、必要例外或降級參考。其中 GLM-4.5 和 OLMo 2 顯式使用 mid-training 術語，GLM-4.5 進一步把 mid-training 拆成 code / reasoning / long-context 三個子階段，是目前公開報告中子階段拆分最細的樣本。

第二，在 Qwen3、OLMo 2、Phi-4、Llama 3 等案例中，late-stage / continued pre-training 往往伴隨更強的資料篩選、能力定向取樣或高質量資料上調；但 DeepSeek-V3 這類 context extension 案例主要支援階段化長度擴充套件，不直接支援資料配比結論。更穩妥的說法是：晚期 token 的邊際價值更依賴質量、配比和訓練目標，而不是簡單追加 web tokens。

第三，長上下文不是隻改 RoPE scaling。真實長上下文能力需要訓練資料、長度 curriculum、位置/注意力機制、推理框架和評估任務共同成立。DCA、YaRN、sparse attention、chunked prefill 等技術要區分其作用位置：有些影響訓練可行性，有些主要服務推理外推和部署成本。

第四，reasoning mid-train 更適合作為“提高 RL 可用 substrate”的強工程假設，而不是已被證明的普遍定律。較保守的說法是：如果 base/mid-trained checkpoint 對目標任務完全沒有可解性，RL 通常很難穩定地產生能力；但 RL 是否帶來真實能力增益，還取決於 pretraining 暴露、任務是否位於模型能力邊界、reward 設計和探索預算。

第五，合成資料有價值，但不是免費午餐。Phi-4 說明 synthetic data 可以顯著影響 reasoning 表現，也提醒團隊仍需要有機資料來補世界知識、表達多樣性和真實分佈。

第六，透明的實驗台賬比單點榜單更重要。OLMo 2 這類報告的最大價值之一，是把 pretraining、mid-training、post-training 和資料構成分開描述，幫助後來團隊建立自己的 lineage、分桶 loss 和迴歸評估。

第七，公開報告能證明“某些成熟團隊採用了階段化 late-stage training”，但很少能證明“某個資料比例、學習率或 token budget 是通用最優”。本文後續所有 recipe 表都應讀作搜尋空間和質量門，而不是從某個模型報告復刻出的配方。

## 3. 公開案例能證明什麼，不能證明什麼

本章不是模型報告複述，也不是 recipe 來源。案例只用於回答“哪些公開事實能轉成內部檢查項”，並遵循版本篩選規則：同一模型家族優先最新一版，最多上一版；更早樣本必須寫明保留理由。

| 案例 / 資料 | 使用級別 | 為什麼保留 | 可安全歸納 | 不得推出 | 內部驗證動作 |
| --- | --- | --- | --- | --- | --- |
| Qwen3 | 主讀：當前/最新公開技術報告樣本 | 披露 General / Reasoning / Long Context 三階段 pre-training | capability shaping 和 context adaptation 可分目標管理 | 原報告自稱 mid-training，或完整資料配比可反推 | 把能力與長上下文實驗拆成不同 experiment_id |
| DeepSeek-V3 | 主讀：DeepSeek-V 系列公開 base report 樣本；若後續同家族新報告披露同類細節，應替換 | 披露 14.8T base pre-training 後的 32K/128K context extension，並強調架構/系統協同 | 長上下文必須同時驗證訓練、架構、serving、eval | context extension 等於普通資料繼續訓練 recipe | 做 offline/online 長上下文一致性和短任務迴歸 |
| GLM-4.5 | 主讀：GLM 家族最新公開技術報告樣本 | 顯式使用 mid-training 術語，披露 ~1.1T tokens 的三子階段（repo-level code / synthetic reasoning / long-context + agent），漸進式 4K→32K→128K context extension | mid-training 可按能力目標拆成多個子階段獨立管理；合成推理資料和 agent 軌跡資料可在 mid-train 階段注入 | GLM-4.5 的 MoE 架構、子階段 token 比例或專家路由策略直接適合 dense model | 按子階段拆 experiment_id；MoE 需額外監控路由熵和專家負載 |

| OLMo 2 | 必要例外 | 明確使用 mid-training 術語，且披露資料混合、micro-annealing、post-training 分層 | mid-train 可獨立管理並保留 lineage/eval gate | OLMo 2 的 Dolmino mix、token 數適合其它模型 | 建立 checkpoint lineage、mid-train scorecard、post-train 前評估 |
| Phi-4 | 必要例外 | 披露 synthetic data for pretraining and midtraining，且顯式 midtraining 承擔 4K -> 16K context 擴充套件 | 合成資料要有 verifier、風格多樣性和有機資料兜底 | reasoning 增益可精確歸因到“合成 CoT mid-train” | 建立 synthetic bucket 驗證、模板重複率和人工錯誤率 |
| Llama 3 | 降級參考 | 僅在需要 late-stage continued pre-training、long-context pre-training、annealing 的公開細節時閱讀 | late-stage 收斂需繫結高質量資料、LR、checkpoint 選擇 | Llama 3 token budget 或 annealing recipe 是小團隊預設值 | 把 annealing 當搜尋變數，不當預設動作 |
| PRISM / pre-mid-RL 研究 | 機制假設，不屬於模型版本線 | 用受控實驗提出 retention、interaction、RL substrate 問題 | 機制假設可進入 proxy ablation | 可升級成產業大模型定律 | 用 paired SFT/RL protocol 驗證收益保留 |

這張表的使用方式很簡單：先讀主讀樣本；只有當前模型報告沒有覆蓋某個關鍵問題，才讀上一版或必要例外。若目標團隊做的是某個特定模型家族，例如 Qwen、DeepSeek、Llama 或自研模型，內部資料包應優先替換成該家族最新報告，再保留最多一份上一版作為對照。

舊版本案例只保留三種情況：第一，它提供當前版本沒有的術語或治理結構，例如 OLMo 2 的 mid-training 分層；第二，它提供專項能力的關鍵工程拆解，例如 Qwen2.5-1M 對 1M context 的訓練/外推/部署拆分；第三，它提供一個仍未被新報告替代的受控變數，例如 Llama 3 的 annealing 或 Phi-4 的 synthetic data 邊界。除此之外，舊模型不要進入主讀清單。

## 4. Mid-train 的設計框架

本章先按第一輪最小閉環來讀：Gate 0 凍結目標能力、不可犧牲能力、eval registry 和 post-train paired protocol；Gate 1 凍結 data manifest；Gate 2 用 control mix + 3-5 個單變數變體在 proxy scale 排除壞方向；Gate 3 用 target model 短程訓練尋找能力/遺忘 Pareto checkpoint；Gate 4 用同一套 frozen post-train paired protocol 驗證收益是否保留。4.1-4.6 分別解釋這個閉環中的目標、資料、訓練旋鈕、ablation、curriculum 和組織質量門。

一個可執行的 mid-train 專案，最終應回答五個問題：目標能力是什麼，哪些能力不能退化，能力由哪些資料桶驅動，訓練長度/學習率/context curriculum 如何設定，以及如何證明它對 post-train 有幫助。

### 4.0 第一輪最小閉環

第一次做 mid-train 時，不要先設計完整訓練系統。這裡的“最小閉環”不是最小訓練配方，而是最小證據鏈：用儘可能小的實驗代價驗證資料、訓練、評估和 post-train 接力是否能形成可解釋結論。先跑通一個目標、一個主假設、一組資料桶、一套 eval registry、一個短程訓練視窗、一個 frozen post-train paired protocol。

| 閉環環節 | 只回答一個問題 | 最小證據 | 失敗就回到哪裡 |
| --- | --- | --- | --- |
| 目標 | 為什麼必須 mid-train | 目標能力、不可犧牲能力、替代方案排除 | 回到 charter，改目標或不做 mid-train |
| 資料 | 哪個桶可能驅動目標能力 | data manifest、汙染檢查、linked eval、人工樣例 | 回到資料桶准入卡 |
| 訓練 | 低風險短程是否出現方向性收益 | 固定 LR/mix/context 的 checkpoint 曲線和分桶 loss | 回到訓練旋鈕或資料 mix |
| 評估 | 收益是否不是公開榜單或格式假陽性 | private holdout、raw output、人工審查、scorecard | 回到 eval registry |
| 接力 | post-train 後收益是否保留 | paired SFT/RL protocol、seed、AUC、失敗樣例 | 回到 post-train recipe 或 mid-train 目標 |
| 決策 | 是否 extend / rollback / kill | Gate dashboard、evidence_path、owner 簽字 | 回到對應 gate |

第 11 章給出的是這個閉環的一組示例搜尋網格，不是預設配方。第 4 章先講設計物件，第 6-8 章講如何判斷證據，第 11 章才給可執行起步樣例。

### 4.1 能力目標分層

建議把目標分成四層：

第一層：通用 base 保真

- 通用語言建模 loss 不異常上升。
- 常識、事實、多語、基礎指令理解不明顯退化。
- 多輪對話雖然不是 mid-train 主目標，但基本可讀性不能崩。

第二層：核心能力增強

- 數學：GSM8K、MATH、AIME 類，以及內部可驗證題庫。
- 程式碼：HumanEval、MBPP、EvalPlus、repo-level completion、FIM、多檔案定位等；SWE-bench 更適合在固定 harness、輕量 instruction wrapping 或 post-train readiness 階段使用。
- STEM：GPQA、MMLU-Pro、領域題庫。
- 推理：BBH、ARC、邏輯推理、複雜 QA。

第三層：結構能力

- 長上下文檢索、綜合、衝突消解。
- FIM、程式碼補全、多檔案編輯。
- 表格、日誌、長 PDF、長會議紀要理解。

第四層：post-train readiness

- SFT 後是否更容易學會目標格式。
- RL/RLVR 後是否更穩定提升。
- reward hacking 是否減少。
- 推理軌跡是否更可控。

### 4.2 資料桶設計

建議至少建立以下資料桶：

| 資料桶 | 目標 | 主要風險 | 建議控制 |
| --- | --- | --- | --- |
| 高質量 web / book | 保持通用語言與世界知識 | 噪聲、重複、版權/合規 | 去重、質量分類、取樣上限 |
| 程式碼 | 程式設計能力、形式化結構 | license、重複、benchmark 洩漏 | license 過濾、repo 去重、題庫隔離 |
| 數學 / STEM | 推理與符號能力 | 過擬合競賽題、解答錯誤 | 可驗證題、難度分層、proof/answer 校驗 |
| 合成教材 | 概念壓縮、教學式推理 | 風格單一、teacher 錯誤 | 多 teacher、多模板、驗證器過濾 |
| QA / CoT | 推理軌跡和任務格式 | 把 post-train 行為提前固化 | 控制比例，保留自然文字 |
| 長文件 | 長上下文適應 | 空洞 padding、低效 token | 真實長文件 + 合成長任務 |
| FIM / 程式碼編輯 | coding agent 能力 | 破壞普通 LM 分佈 | 分桶 loss 和程式碼 eval |
| 領域資料 | 行業/產品能力 | 災難性遺忘、狹窄化 | 低比例混入，通用迴歸 |

不要只做一個全域性 mix。每個桶要有：

- 資料來源和 license 記錄。
- 去重策略。
- 質量打分。
- contamination 檢查。
- 取樣比例。
- 單獨 loss。
- 對應 eval。

### 4.2.1 資料桶准入卡

每個資料桶進入訓練前，都應先進入 data manifest，而不是直接進入 mix。最小欄位如下：

| 欄位 | 說明 |
| --- | --- |
| `bucket_id` / `bucket_name` | 資料桶編號和名稱 |
| `source_manifest` | 原始來源、時間範圍、license / 合規狀態 |
| `processing_commit` | 清洗、解析、過濾指令碼版本 |
| `dedup_version` | 文件級、段落級、程式碼 repo 級、題目級去重版本 |
| `contamination_version` | 與 public benchmark、private holdout、合成 prompt 的汙染檢查版本 |
| `tokenizer_version` | tokenizer 和 special tokens 版本 |
| `quality_score_distribution` | 質量打分分佈，不只寫平均分 |
| `language_domain_distribution` | 語言、領域、檔案型別、長度分佈 |
| `time_distribution` | 資料時間範圍、新舊分佈、是否包含訓練後新事件 |
| `sampling_weight` | 當前 mix 中取樣權重和上限 |
| `shuffle_seed` / `bucket_interleaving` | 資料順序、bucket 交織策略、是否階段性改 mix |
| `bucket_validation_loss` | 分桶 validation loss 和異常變化 |
| `linked_eval_id` | 對應 eval registry 中的評估項 |
| `known_risks` | license、汙染、模板化、重複、事實錯誤、分佈過窄 |

資料桶准入門建議寫成硬規則：

> [!CAUTION]
> - **沒有 manifest/hash，不進入訓練。**
> - **沒有對應 eval，不提高取樣權重。**
> - **汙染命中未處理，不進入 primary 實驗。**
> - **合成資料沒有正確性過濾和風格多樣性檢查，不超過低比例探索。**
> - **領域資料沒有通用迴歸保護，不進入主模型短程。**
> - **長上下文資料只有拼接文字、沒有真實長任務，不進入 long-context primary mix。**
> - **任何改過濾器、質量模型、dedup 閾值、取樣權重、合成生成 prompt 或資料順序的動作，都必須生成新的 `data_manifest_hash` 和 `experiment_id`。**

### 4.2.2 合成資料使用邊界

合成資料需要單獨治理，因為它最容易在短期指標上“看起來很有效”：

| 風險 | 表現 | 控制方式 |
| --- | --- | --- |
| Teacher 錯誤放大 | 數學/程式碼答案看似流暢但不可驗證 | 可執行驗證、答案校驗、抽樣人工複核 |
| 風格模板化 | 輸出都像同一個 teacher，推理步驟固定 | 多 teacher、多 prompt、多解法、混入有機資料 |
| Benchmark 近似汙染 | 公開題變體大量出現 | 題幹、解法結構、embedding 和 translated duplicate 檢查 |
| 難度分佈失真 | 訓練題過簡單或只覆蓋一種題型 | 難度分桶、失敗樣例回灌、私有 holdout 對照 |
| CoT 行為前移過度 | base 模型過早學會冗長解釋格式 | 控制 CoT 比例，混入短答、無 CoT、自然文字 |

**合成資料的合理用法是”可驗證能力放大器”，不是低成本 token 替代品。**數學、程式碼、結構化推理適合多用可驗證合成；開放域知識、語言風格、多語表達仍需要高質量有機資料兜底。

不同合成資料型別的邊界也不同：

| 合成資料型別 | 適合用途 | 不適合用途 | 必須檢查 |
| --- | --- | --- | --- |
| 可驗證數學/程式碼 | 提高符號推理、可執行解題、單元測試能力 | 替代開放域知識 | verifier、答案執行、難度分桶 |
| 教材式解釋 | 壓縮概念、補齊知識結構 | 大比例替代自然語料 | teacher 多樣性、表達模板、事實抽檢 |
| CoT / rationale | 提供推理軌跡形式 | 讓 base 過早固化冗長回答風格 | CoT 佔比、短答混入、風格漂移 |
| 長上下文合成任務 | 覆蓋稀缺長任務模式 | 替代真實長文件和長程式碼 | 證據位置、干擾資訊、不可回答樣本 |
| 領域合成 QA | 補足領域任務形態 | 替代真實領域文件 | 專家抽檢、來源追蹤、術語一致性 |

Synthetic bucket 還必須記錄 teacher model/version、生成 prompt、取樣引數、verifier 版本、過濾規則、被拒樣本比例、模板重複率、答案可驗證率、人工抽檢錯誤率，以及生成器是否讀過 eval/benchmark。合成 CoT/QA 要額外檢查 style entropy、template n-gram 和 response length distribution，避免把 post-train 行為格式提前固化。

### 4.3 學習率和 token budget

公開報告通常不會完整給出所有細節，但工程上有幾條穩健原則。

第一，mid-train 通常不應使用接近 pretrain 初期的高學習率。模型已經形成大量通用能力，過高學習率會導致遺忘、風格漂移和 loss spike。

第二，學習率策略要和目標匹配：

- 短程能力增強：低學習率繼續訓練，觀察專項能力和通用迴歸。
- late-stage annealing：進一步降低學習率，用高質量資料收斂。
- context extension：學習率更保守，並配合長度 curriculum。
- domain adaptation：更要限制 token budget 和資料比例，避免模型變窄。

第三，token budget 不是越大越好。下面三檔只是啟動實驗的保守刻度，不按模型規模線性通用，也不能替代 scaling ablation：

- 1B-10B tokens：用於 pipeline、資料質量、LR、汙染檢查和 proxy ablation。
- 10B-100B tokens：作為可觀察能力增強和中小模型 recipe 驗證的示例區間，實際預算要由模型規模、資料質量和 gate 結果決定。
- 100B-1T+ tokens：才適合主模型較完整的 capability shaping 或 long-context extension。

以上不是行業定律，而是工程上較保守的起步刻度。真正 token budget 應由模型規模、資料質量、目標能力和預算共同決定。

### 4.3.1 配方旋鈕與不可同時改變原則

訓練變數也要像資料和 eval 一樣凍結版本。每次 run 至少記錄：

| 變數 | 為什麼要凍結 | 常見誤判 |
| --- | --- | --- |
| `base_checkpoint_hash` | 決定起點能力和已有資料暴露 | 把不同 base 的差異誤判成 mid-train 收益 |
| `optimizer_state_strategy` | 接續或重置會改變 loss 曲線和穩定性 | 重置後 warmup 不足導致假性退化 |
| `lr_schedule` / `warmup` | 影響遺忘、收斂和 loss spike | 只比較最終分數，不比較訓練穩定性 |
| `peak_lr` / `min_lr` / `decay_shape` | 決定實際最佳化軌跡 | 都叫“低 LR”，但訓練動態完全不同 |
| `gradient_clipping` / `precision` | 影響穩定性和數值誤差 | 把數值不穩誤判成資料 mix 問題 |
| `global_batch` / `microbatch` | 改變有效 batch 和訓練噪聲 | 小 batch 噪聲被誤判成能力波動 |
| `sequence_length_distribution` / `length_bucket_ratio` | 改變有效 token 分佈和訓練動力學 | 把長度變化帶來的收益歸因給資料 mix |
| `data_order` / `shuffle_seed` | curriculum、annealing、長上下文階段對順序敏感 | 同一 mix 不同順序得出相反結論 |
| `packing_policy` / `masking_policy` | 影響 FIM、長文件和短樣本比例 | 短任務退化但全域性 loss 正常 |
| `checkpoint_selection_policy` | 決定最終比較物件 | 事後挑最佳 checkpoint 誇大收益 |
| `eval_cadence` | 決定何時發現遺忘和漂移 | 訓練結束才發現不可逆回撤 |

Scale-up 也要保守。Proxy 成功只能說明某個方向值得繼續，不說明目標規模必然成功。進入更大訓練前至少滿足四個條件：

1. 兩個 scale 或 target 短程 run 的方向一致。
2. 分桶 loss、目標 eval、base regression 的變化能被同一個資料/訓練假設解釋。
3. 相同 mix 下不同 seed 不反轉，訓練吞吐、checkpoint resume 和監控已經驗證。
4. 成本上可接受：如果放大 10 倍 token 後只可能換來邊際小幅收益，就應先改善資料或 eval，而不是加預算。

一次實驗不要同時改太多旋鈕。建議把常見旋鈕拆開看：

| 旋鈕 | 可調方式 | 主要風險 | 觀測訊號 | 何時停止 |
| --- | --- | --- | --- | --- |
| LR / warmup | 低 LR、進一步降低、短 warmup sweep | 遺忘、loss spike、收斂過慢 | 全域性 loss、分桶 loss、梯度範數 | spike 不恢復或保真指標快速下跌 |
| Token budget | 遞增短程預算 | 浪費預算、過擬合窄分佈 | checkpoint 間 delta、Pareto 曲線 | 邊際收益消失或回撤擴大 |
| Sequence length | 逐步拉長長度分佈 | 長訓傷短任務、吞吐下降 | 長度分桶 loss、短任務迴歸 | 短任務顯著退化或 serving 成本不可接受 |
| Batch / packing | 改 packing、混長短樣本 | 有效資料分佈改變 | 吞吐、padding、短樣本 loss | 全域性 loss 正常但分桶異常 |
| Optimizer state | 接續或重置 | 繼承動量或 warmup 不足 | early loss、梯度、resume 連續性 | early instability 無法解釋 |
| Synthetic ratio | 逐步增加上限 | 模板化、teacher 錯誤放大 | 私有 eval、風格審查、失敗樣例 | 風格坍縮或私有集不漲 |
| Domain ratio | 低比例混入 | 模型變窄、通用能力退化 | 通用迴歸、領域私有任務 | 領域漲幅小於通用回撤代價 |

### 4.4 第一輪 ablation 設計表

第一輪不要追求“最好配方”，而要快速排除錯誤方向。下表中的比例只是搜尋變數示例，用來暴露靈敏度和壞方向，不能作為主訓練預設 mix。

| 實驗目的 | 模型規模示例 | Token budget 示例 | 資料 mix 搜尋變數示例 | LR 變數 | 成功指標 | 停止條件 |
| --- | --- | --- | --- | --- | --- | --- |
| Pipeline 和 eval 校準 | proxy model，例如 1B/3B | 1B-3B tokens | 小樣本全桶 mix | 低 LR 單點 | loss 正常、eval 可復現 | loss spike、eval 不穩定 |
| 通用保真底線 | proxy model，例如 3B/7B | 3B-10B tokens | 通用高質量佔比 30/50/70% | 固定低 LR | 通用能力回撤可控 | 多語/事實大跌 |
| 數學/STEM 注入 | proxy model，例如 3B/7B | 5B-20B tokens | 數學/STEM 10/20/30% | 低 LR + 更低 LR | 數學提升且通用不崩 | 只提升公開題型 |
| 程式碼/FIM 注入 | proxy model，例如 3B/7B | 5B-20B tokens | 程式碼/FIM 10/20/30% | 固定低 LR | completion/FIM/repo task 提升 | 普通生成明顯程式碼化 |
| 合成資料佔比 | proxy model，例如 3B/7B | 5B-20B tokens | synthetic 5/15/30% | 固定低 LR | 可驗證任務提升、風格多樣 | 模板化或汙染風險 |
| 主模型短程驗證 | target model | 10B-100B tokens | 候選 2-3 個 mix | 低 LR sweep | Pareto checkpoint 出現 | 目標能力不漲或遺忘過大 |

這些實驗的產出不是最終模型，而是決策：哪個資料桶有效，哪個比例危險，哪個學習率區間安全，是否值得進入更長主訓練。

### 4.4.1 如何讀 ablation

Ablation 的產出是排除壞方向和形成候選 Pareto，不是證明最終配方。建議按五條規則判讀：

1. **Proxy 成功不等於 target 成功**；進入主模型前需要兩個 scale 方向一致，或 target 短程 run 複核。
2. **單一公開榜上漲不等於能力增強**；必須和 private holdout、失敗樣例簇、人工審查一起看。
3. **Target 能力提升必須和 guardrail 一起看**；數學/程式碼漲而多語、事實、自然生成明顯跌，不算乾淨收益。
4. 接近閾值的結果要重複 run 或相鄰 checkpoint 複核；**不要用一次 seed 的小幅收益做放大決策**。
5. **任何中途改 mix、filter、LR、長度 curriculum 或 eval 版本的實驗，都必須重開 experiment_id。**

### 4.5 Curriculum：先能力，後長度，還是並行？

建議新團隊不要一開始把所有目標混在一起。

更穩的順序是：

1. Base 保真檢查：確認 checkpoint、tokenizer、optimizer state、資料 pipeline、評估體系可用。
2. 能力型 mid-train：數學/程式碼/STEM/高質量合成資料，短程 ablation。
3. 長上下文適應：從較短 context 逐步拉長，單獨監控長上下文 loss 和真實任務。
4. 小規模 post-train 接力：用固定 SFT/RL recipe 比較 mid-train 前後 checkpoint。
5. 決定是否合併為主訓練配方。

為什麼建議先能力後長度？因為長上下文訓練成本高、debug 難，而且容易把架構和資料問題混在一起。如果模型基礎 reasoning/程式碼能力沒有改善，長上下文只是放大輸入視窗，不一定放大任務能力。

也有例外：如果產品目標就是法律/金融長文件 QA 或程式碼庫 agent，長上下文可以前置，但仍建議先在小模型上驗證。

| 產品主目標 | 推薦先驗 | 第一輪驗證 | 反證訊號 |
| --- | --- | --- | --- |
| 數學、程式碼、STEM、複雜 QA | 先能力，後長度 | 能力桶 + 通用保真桶短程 ablation | 目標能力只在 public benchmark 漲，private/人工樣例不漲 |
| 法律/金融/科研長文件 QA | 能力和長度並行，但分兩個實驗 ID | 一條能力短程、一條 32K/64K context 短程 | 長上下文檢索好但綜合/衝突/不可回答任務差 |
| 程式碼庫 agent / repo-level editing | 程式碼/FIM 與中長上下文並行 | FIM/repo task、長程式碼定位、短程式碼迴歸 | repo task 漲但普通程式碼生成或短任務明顯退化 |
| 領域繼續訓練 | 先低比例領域，再看是否需要長度 | 領域桶 5/10/20% + 通用迴歸 | 領域任務漲幅小於通用能力回撤代價 |
| 長上下文展示能力 | 先確認產品價值，再做 context extension | 真實長任務、serving 成本、短任務迴歸 | NIAH 好看但真實任務、線上成本或短任務不成立 |

長上下文 curriculum 要記錄長度分佈，而不是隻記錄最大長度。每個階段至少記錄：

- 4K/16K/32K/64K/128K 樣本比例。
- packing 策略、padding 浪費、短樣本是否被過度上取樣。
- 長樣本來源：自然長文件、長程式碼、合成長任務、跨文件集合。
- 任務結構：證據密度、不可回答樣本、衝突證據、多跳證據、證據位置、輸出長度。
- 訓練 kernel 與推理 kernel 是否一致。

否則“擴到 128K”只是一個最大長度宣告，不能解釋模型到底學到了長上下文理解、長距離檢索，還是只學會在更長輸入裡保持 loss 不崩。

### 4.6 組織執行機制：角色、台賬和質量門

Mid-train 專案不是訓練團隊單獨完成的模型續訓，而是 data、training、eval、post-train 四方共用同一張實驗台賬、同一組質量門的能力塑形專案。每個組織概念都必須繫結工程物件：checkpoint、data manifest、eval version、gate、owner 和 decision。

第一，設定 DRI，而不是泛泛“多團隊參與”。

| 工作流 | 唯一 DRI | 參與團隊 | 最終拍板 | 關鍵產出 |
| --- | --- | --- | --- | --- |
| 目標定義 | Model Lead | Product/Domain、Eval、Post-train | Model Lead | Mid-train charter、不可犧牲能力、實驗假設 |
| 資料構建 | Data DRI | Research、Safety/Compliance、Eval | Data DRI + Safety/Compliance | 資料桶說明、data manifest/hash、lineage、contamination report |
| 訓練執行 | Training Infra DRI | Model、Data、Infra | Training Infra DRI | 訓練 runbook、LR/token budget 配置、checkpoint ledger、事故記錄 |
| 評估體系 | Eval DRI | Model、Product/Domain、Post-train | Eval DRI | eval registry、私有 holdout、scorecard、人工審查樣例 |
| Post-train 接力 | Post-train DRI | Eval、Safety、Model | Post-train DRI | frozen post-train paired protocol、readiness report、對齊迴歸 |
| 長上下文系統 | Infra DRI | Training、Eval、Serving | Infra DRI + Eval DRI | kernel/serving 一致性報告、KV cache/latency/cost 評估 |
| 風險與釋出 | Safety/Compliance DRI | Data、Product、Release | Decision owner | 合規審查、風險登記表、go/no-go memo |

第二，實驗台賬要有固定 schema。沒有台賬的實驗不進入結論；任何中途改 mix、改過濾器、改 eval 版本，都必須新建 experiment_id。

| 欄位 | 說明 |
| --- | --- |
| `experiment_id` / `hypothesis` / `owner` | 實驗編號、要驗證的假設、唯一負責人 |
| `base_checkpoint_hash` / `training_code_commit` | 起點 checkpoint 和訓練程式碼版本 |
| `data_manifest_hash` / `data_bucket_ratio` | 資料 manifest、取樣權重、資料桶比例 |
| `dedup_version` / `contamination_version` | 去重、近重複和汙染檢查版本 |
| `lr_schedule` / `warmup` / `optimizer_state_strategy` | 學習率、warmup、optimizer state 接續或重置 |
| `token_budget` / `sequence_length_distribution` | token 預算和長度分佈，不只記錄最大長度 |
| `eval_suite_version` / `private_holdout_version` | eval registry、私有 holdout、harness commit |
| `seed` / `cost` / `incident_log` | 隨機種子、訓練成本、異常事件和處理動作 |
| `result_summary` / `decision` / `signoff` | 結論摘要，kill/retry/extend/promote 決策，簽字人 |
| `post_train_readiness_result` | 固定 SFT/RL recipe 後收益是否保留 |

第三，質量門的唯一權威看板放在第 8.0 節；本節只定義角色、台賬欄位和簽字責任。本文不能替團隊設閾值，但 gate memo 至少要定義：目標能力最低提升、通用能力最大回撤、私有 eval 透過率、汙染命中率上限、loss spike 處理規則、風格漂移判定、post-train 後收益保留比例。任何 gate 狀態、證據路徑、版本 hash 和過期時間，都以第 8.0 節 dashboard 為準。

| Gate | 進入條件 | 退出條件 | 阻斷條件 | 簽字人 |
| --- | --- | --- | --- | --- |
| Gate 0: charter | 目標能力、不可犧牲能力、預算和 post-train 接力需求明確 | Mid-train charter 透過；v1 eval registry 和 v1 post-train paired protocol 狀態以第 8.0 節 dashboard 為準 | 無 DRI、無 paired protocol owner/schema、無私有 holdout owner/schema | Model Lead、Eval DRI、Post-train DRI |
| Gate 1: 資料凍結 | 資料來源、license、清洗、去重和汙染檢查完成 | data manifest/hash 凍結，人工抽檢無系統性錯誤 | benchmark 洩漏、license 不清、合成資料模板化明顯 | Data DRI、Safety/Compliance DRI、Eval DRI |
| Gate 2: proxy ablation | 至少兩個候選 mix 和固定 eval suite | proxy 只用於排除壞方向；進入主模型前需兩個 scale 方向一致，或 target 短程驗證關鍵訊號 | 只漲公開 benchmark，私有任務不動，通用能力大跌 | Model Lead、Eval DRI、Data DRI |
| Gate 3: 主模型短程 | 訓練 runbook、resume、監控和 kill switch 就緒 | 出現能力/遺忘 Pareto checkpoint，分桶 loss 可解釋 | NaN、不可恢復 loss spike、梯度異常、吞吐異常、resume 不連續、長上下文傷短任務 | Training Infra DRI、Eval DRI、Model Lead |
| Gate 4: post-train 接力 | paired protocol 的 recipe、seed、預算、harness、失敗判定已凍結 | 固定 paired protocol 下 post-train 後收益仍保留，安全/風格無顯著惡化 | mid-train 收益在 post-train 後消失、paired protocol 中途改變，或對齊難度顯著增加 | Post-train DRI、Eval DRI、Safety DRI |
| Gate 5: 放大訓練 | 收益來源可解釋，資料/eval/訓練自動化穩定 | go/no-go memo 透過，進入下一階段預算 | 收益依賴汙染、窄評估、偶然 checkpoint 或無法復現 | Decision owner、Model Lead、Release owner |

第四，會議節奏要服務決策，不服務彙報：

- Daily training review：loss、吞吐、NaN/loss spike、checkpoint、resume、事故處理。
- Twice-weekly eval review：目標能力、通用迴歸、私有 holdout、人工樣例審查。
- Weekly decision review：kill、retry、extend、promote，只基於台賬和 gate memo 決策。
- Post-train handoff review：paired protocol 是否有效，收益是否保留，安全和風格是否退化。

這套機制的底線是：**日曆不能替代質量門**。90 天只是參考節奏，是否進入下一階段由資料就緒度、訓練穩定性、eval 穩定性和 post-train 接力驗證決定。

## 5. 訓練工程檢查點

### 5.1 啟動前檢查

啟動 mid-train 前，應完成：

- Base checkpoint 的完整 eval 快照。
- Optimizer state 是否接續的決策。
- Tokenizer 和 special tokens 凍結。
- 資料去重和 benchmark contamination 檢查。
- 每個資料桶的樣例審查。
- 小規模 dry run，確認 loss、吞吐、視訊記憶體、checkpoint、resume 正常。
- Post-train DRI 確認後續 SFT/RL 的 paired protocol。

如果沒有 post-train 接力驗證，mid-train 很容易最佳化錯目標。

### 5.2 訓練中監控

至少需要監控：

- 全域性 training loss / validation loss。
- 按資料桶分解的 validation loss。
- 通用能力 benchmark 的迴歸。
- 目標能力 benchmark 的增益。
- 長上下文不同長度段的 loss。
- 生成樣例的風格漂移。
- MoE 模型的專家負載、路由熵和 token drop。
- 梯度範數、loss spike、checkpoint 間差異。

**不要等訓練結束才做 eval。** Mid-train 的失敗常常在早期就有訊號：通用 eval 大跌、輸出風格模板化、某個資料桶 loss 異常、長上下文短任務退化。

### 5.3 Checkpoint 策略

建議保留：

- pre-mid checkpoint。
- 每個階段的 early/mid/final checkpoint。
- 學習率變化前後的 checkpoint。
- context length curriculum 每個長度階段的 checkpoint。
- 用於 post-train 接力的候選 checkpoint。

不要只保留最後一個。**Mid-train 的最佳 checkpoint 可能不是 loss 最低的 checkpoint，而是目標能力和遺忘之間的 Pareto 最優點。**

### 5.4 容易低估的工程風險

第一，mixed-length batching 和 context parallelism 會帶來負載不均。長上下文 batch 的吞吐、視訊記憶體碎片、padding 浪費和通訊模式都和短上下文不同，訓練監控必須按長度段拆開，否則全域性吞吐會掩蓋區域性不穩定。

第二，optimizer state 接續或重置都可能出問題。接續 optimizer state 更像原訓練軌跡延長，但可能繼承舊階段動量；重置 optimizer state 更乾淨，但需要重新設計 warmup，否則容易出現 loss spike 或過慢收斂。兩種方案都應先在 proxy scale 測。

第三，**訓練 attention kernel 和部署 attention kernel 不一致會製造假陽性**。離線 eval 透過，不代表線上 128K/1M 服務可用；長上下文訓練、推理外推、kv cache、chunked prefill、sparse attention 和 serving 框架必須聯合驗證。

## 6. 評估體系

### 6.0 從 benchmark list 到 eval registry

Mid-train 的評估系統不是榜單集合，而是一組帶版本、汙染狀態、執行 harness、閾值和原始輸出存檔的 eval object；**沒有進入 registry 的評估結果，只能作為觀察，不能作為 go/no-go 依據**。

每個專案最多設定 3-5 個 primary 指標，另設若干 guardrail 和 diagnostic。Primary 太多會誘導 cherry-pick，diagnostic 太少會讓團隊不知道為什麼漲跌。

### 6.1 Eval registry 的最小 schema

Eval DRI 應在 Gate 0 凍結 v0 owner 和 schema，並在進入 Gate 1 前凍結 v1 eval registry。最小欄位如下：

| 欄位 | 含義 |
| --- | --- |
| `eval_id` | 唯一編號 |
| `eval_family` | base regression / target capability / long-context / post-train readiness / safety-style |
| `task_name` | 任務名 |
| `data_source` | public / dev-private / sealed-private / final-blind / synthetic / human-written |
| `created_at` / `data_hash` | 建立時間和資料 hash；私有集必須早於對應訓練資料 mix 凍結 |
| `split_policy` | dev / private holdout / regression set / final-blind |
| `contamination_check_version` | exact / n-gram / MinHash / embedding / paraphrase / translated duplicate / manual |
| `harness_commit` | 評估程式碼版本 |
| `prompt_or_wrapper_version` | base prompt、minimal prompt、few-shot policy、light instruction wrapper、agent harness 版本 |
| `decoding_params` | temperature、top_p、max tokens、pass@k 等 |
| `metric` / `aggregation` | accuracy、pass@k、F1、judge score、latency/cost 等，以及聚合方式 |
| `decision_role` | primary / guardrail / diagnostic |
| `threshold_or_guardrail` | 提升閾值、最大回撤、失敗條件、最小顯著差異 |
| `sample_size` | 樣本量和抽樣規則 |
| `manual_review_rubric` | 需要人工審查時的 rubric |
| `raw_output_path` | 原始輸出存檔位置 |
| `owner_note` | 解釋性備註，不放組織責任 |

Eval 版本變更規則要寫死：改樣本、改 prompt、改答案抽取器、改 metric、改 harness、改閾值，都必須生成新 `eval_suite_version`，舊結果不能橫向混比。

私有 holdout 建議分三層：

- dev-private：調 pipeline、校準 harness，不進入最終 gate。
- sealed-private：階段門使用，禁止針對它調資料 mix 或 prompt。
- final-blind：只在 Gate 5 或釋出前使用，避免被反覆最佳化成“另一套訓練目標”。

### 6.2 Eval suite matrix

| Eval family | 目的 | 代表任務 | 適用 checkpoint | Metric | 失敗訊號 | Raw output 要求 |
| --- | --- | --- | --- | --- | --- | --- |
| Base regression | 保護不可犧牲能力 | 通用知識、多語、基礎生成、事實 QA | pre-mid、mid、post-train 後 | 最大回撤、低分桶變化、人工審查 | 平均分小跌但某語言/風格崩 | 固定樣本籃子完整輸出 |
| Target capability | 判斷目標能力是否真的變強 | 數學、程式碼、STEM、領域任務、FIM | pre-mid、mid | delta、難度分桶、私有任務透過率 | 只漲公開榜，私有任務不動 | 成功/失敗樣例簇 |
| Long-context | 判斷長上下文是否可用 | 多證據 QA、衝突消解、長程式碼定位、長表格/日誌 | mid、post-train 後 | 長度分桶表現、短任務迴歸、latency/cost | needle 滿分但真實任務差 | 輸入長度、證據位置、backend 配置 |
| Post-train readiness | 判斷收益是否在 SFT/RL 後保留 | 固定 SFT/RL recipe 後的真實任務和安全評估 | paired post-train checkpoints | 收益保留比例、訓練曲線 AUC、方差 | base 漲但 post-train 後消失 | recipe、seed、checkpoint selection policy |
| Safety/style | 防止風格、安全和拒答漂移 | 安全集、拒答邊界、中文寫作、長答案質量 | mid、post-train 後 | 迴歸率、人工 rubric、拒答誤傷 | 模板化、冗長 CoT、幻覺變多 | 人工審查樣本和評分理由 |

目標能力 eval 回答“能力是否真的變強”；base regression eval 回答“有沒有傷到別的能力”；post-train readiness eval 回答“這個 base 變化是否能穿過 SFT/RL 仍然有價值”。

### 6.3 Base regression eval

Base regression 的重點不是證明模型更強，而是保護不可犧牲能力。它應覆蓋：

- 通用知識：MMLU、MMLU-Pro、內部知識題。
- 推理：BBH、ARC、GPQA、邏輯題。
- 數學：GSM8K、MATH、AIME 風格內部題。
- 程式碼：HumanEval、MBPP、EvalPlus、repo-level completion、FIM、多檔案定位。
- 多語：中文、英文和目標市場語言。
- 生成質量：長回答、摘要、解釋、翻譯、自然對話。

Base model 未必穩定遵循複雜格式，所以要區分“能力不足”和“格式不服從”。每個 base eval 應記錄 minimal prompting、few-shot policy、答案抽取器版本和人工複核規則。SWE-bench、agent 任務、tool-use eval 混入了指令遵循、工具協議、環境穩定性和 scaffold 質量，更適合作為 readiness/diagnostic，而不是裸 base 主指標。

Base regression scorecard 至少報告：

- 不可犧牲能力最大回撤，而不只是平均分。
- 低分桶和長尾語言變化。
- 輸出風格、拒答、冗長程度和幻覺樣例。
- 固定人工審查樣本籃子的結論。

### 6.4 Target capability eval

Target capability eval 要和 base regression 分開。它不只是問“公開榜有沒有漲”，而是問“目標能力是否在真實任務和私有任務上變強”。

| 目標能力 | 評估方式 | 必要防線 |
| --- | --- | --- |
| 數學/STEM | 難度分桶、可驗證答案、私有變體題、失敗樣例簇 | 防止競賽題近重複和 synthetic overfit |
| 程式碼/FIM | HumanEval/MBPP/EvalPlus、FIM、repo-level completion、多檔案定位 | SWE-bench/tool-use 作為 readiness 或 diagnostic |
| 領域能力 | 領域私有題、真實文件 QA、專家人工 rubric | 防止領域資料讓通用能力變窄 |
| Agent/tool-use | 固定 harness、環境版本、工具協議、任務軌跡 | 不把 scaffold 變好誤判成 base 變強 |

如果目標能力只在 public benchmark 上升，而 private holdout、失敗樣例簇和人工審查不支援，應先視為汙染、格式適配或窄分佈最佳化。

### 6.5 Long-context eval

長上下文評估要分三層：

- 機械檢索：Needle-in-a-haystack、多 needle、不同位置和不同噪聲密度。它只能作為 diagnostic，不應進入主 gate。
- 真實理解：長文件 QA、多章節摘要、多證據綜合、衝突資訊判斷、長表格/日誌分析。
- 任務完成：長程式碼庫修改、多檔案 bug 定位、法律/財報/論文綜合分析、agent 軌跡覆盤和下一步決策。

長上下文 eval registry 必須按軸展開：

| 軸 | 必測切片 |
| --- | --- |
| Context length bucket | 4K、16K、32K、64K、128K；如目標是 1M，再加 256K/1M |
| 任務型別 | 檢索、摘要、多證據 QA、程式碼定位、跨段推理、不可回答問題 |
| 證據密度 | 稀疏事實、密集表格、長程式碼、衝突證據 |
| 干擾資訊 | 無干擾、同主題干擾、矛盾證據、過期資訊 |
| 證據位置 | 開頭、中間、末尾、多位置證據 |
| 輸出要求 | 短答、結構化表格、長答案、引用證據 |
| 推理 backend | attention kernel、KV cache、chunked prefill、batching、serving 配置 |
| 成本約束 | latency、吞吐、視訊記憶體、單位請求成本 |
| 短任務迴歸 | 短任務放進長上下文後是否退化 |

長上下文的 hard gate 不應只看“最長長度能跑通”。更合理的門檻是：短上下文能力不明顯退化，中長上下文真實任務有收益，推理系統在目標長度下成本和延遲可接受。

### 6.6 Post-train readiness eval

Post-train readiness 要用 paired protocol：

- pre-mid checkpoint 和 mid checkpoint 使用同一套 frozen post-train paired protocol。
- 固定資料順序、seed、預算、checkpoint selection policy、reward/verifier、harness、decoding 和失敗判定。
- 關鍵結論接近閾值時，跑多 seed 或重複 run，並報告方差或 bootstrapped CI。
- 不用“最佳 post-train checkpoint”做事後比較；要比較固定預算、固定選擇規則下的結果，或報告訓練曲線 AUC。

這裡凍結的 SFT/RL recipe 是診斷儀器，用於比較 pre-mid 與 mid checkpoint，不代表最終生產 post-train 配方。若為追求更好結果而調整 recipe，應重新登記 experiment_id，原 mid-train 結論不得直接沿用。

| 比較項 | 看什麼 | 失敗訊號 |
| --- | --- | --- |
| 收斂速度 | 同預算下是否更快達到目標能力 | 只在個別 seed 變快 |
| 最終能力 | 目標能力是否在 post-train 後仍保留 | base 漲，post-train 後消失 |
| Reward 行為 | RL 是否減少 reward hacking 或提升可驗證任務 | 冗長推理、投機格式、答案漂移 |
| 安全/風格 | 拒答邊界、中文表達、長答案質量 | 安全迴歸、模板化、拒答誤傷 |
| 長上下文 | post-train 後長任務是否繼續可用 | SFT 後長上下文能力被覆蓋 |

四格判讀：

| 結果 | 解釋 | 決策 |
| --- | --- | --- |
| base 漲，post-train 後也漲 | 最可信的 mid-train 收益 | 可進入放大候選 |
| base 漲，post-train 後不漲 | 可能是 base eval 假陽性或 post-train 覆蓋掉收益 | 回到資料/eval，不能直接放大 |
| base 不漲，post-train 後漲 | 可能改善了可訓練性或與 SFT/RL 互動 | 重複 seed 和真實任務驗證 |
| base 跌，post-train 後漲 | 高風險 trade-off | 只有不可犧牲能力未破壞時才可繼續 |

### 6.7 Scorecard：如何讀評估結果

每個候選 checkpoint 都應生成一頁 scorecard。它是 Eval DRI 給 gate 的輸入，不替代 4.6 的最終 go/no-go 決策。

| 欄位 | 內容 |
| --- | --- |
| `checkpoint_id` | checkpoint hash、訓練步數、token 數、context length |
| `target_capability_delta` | primary 指標變化、難度分桶變化、私有任務變化 |
| `base_regression_max_drawdown` | 不可犧牲能力最大回撤和低分桶變化 |
| `private_holdout_result` | dev-private、sealed-private 的使用情況和結果；final-blind 在 Gate 5 或釋出前應保持 unused/reserved，並記錄是否仍未觸碰 |
| `long_context_result` | 長度分桶、真實任務、短任務迴歸、serving 配置 |
| `post_train_readiness_result` | paired protocol、seed、收益保留、方差 |
| `contamination_status` | exact/near-duplicate/embedding/manual 檢查結果 |
| `known_regressions` | 已知退化、失敗樣例簇、人工審查結論 |
| `recommended_action` | kill / retry / extend / promote |

**如果某個指標收益異常高，而私有任務、人工審查和相鄰 eval 不支援，應先假設汙染、格式適配或過擬合，而不是先宣佈能力突破。**

Gate meeting 裡先看一行紅黃綠摘要，再看完整 scorecard：

| 狀態 | 含義 | 會議動作 |
| --- | --- | --- |
| Green | target、guardrail、private holdout、post-train readiness 同方向支援，且證據路徑完整 | 可討論 extend/promote |
| Yellow | 有收益但方差、回撤、汙染、post-train 保留或 serving 成本仍不清楚 | 指定 owner 補證據，不進入放大 |
| Red | 不可犧牲能力越線、汙染/license 未清、事故未歸因、post-train 後系統性變差 | rollback/kill，或重開 experiment_id |

## 7. 風險與事故處理 Runbook

第 7 章不是失敗模式科普，而是訓練中的事故處理 runbook：任何異常都必須能落到一個症狀、一個實驗 ID、一個受影響 checkpoint、一組證據路徑和一個繼續/回滾/kill 決策。

### 7.0 如何使用本章

正常監控、eval registry 和 gate 機制分別放在第 4-6 章；本章只處理異常。觸發異常後，值班 DRI 先凍結現場，再做 30 分鐘首輪診斷，最後給出臨時決策。沒有完成歸因前，不要在同一個 run 裡改資料、改學習率、改 eval harness 或改 post-train recipe。

事故出現後先做三步，不要先改配置：

| 步驟 | 動作 | 輸出 |
| --- | --- | --- |
| 1. 凍結現場 | 儲存 checkpoint、optimizer、data manifest、eval raw outputs、系統指標和相關 commit | `incident_id` 和 evidence paths |
| 2. 判定級別 | 判斷 observation / alert / blocker，指定 30 分鐘首查 owner | 臨時 stop/continue 決策 |
| 3. 分層隔離 | 依次做 system replay、data manifest diff、eval harness diff、model checkpoint diff | root cause 或 unresolved 標記 |

事故先按六類登記：

| 事故型別 | 典型範圍 |
| --- | --- |
| numerical | NaN、Inf、loss 爆炸、梯度異常、混合精度溢位 |
| data | 資料桶汙染、格式漂移、license 問題、合成資料模板化、取樣權重錯誤 |
| eval | harness、prompt、judge、metric、抽樣或汙染導致的分數異常 |
| infra | 吞吐、GPU 利用率、NCCL、儲存、checkpoint、serving 配置異常 |
| capability | 目標能力不漲、通用能力回撤、guardrail 變壞、長上下文真實任務退化 |
| post-train handoff | SFT/RL 接力後收益消失、reward hacking 增加、recipe 噪聲放大 |

> [!CAUTION]
> **硬規則：任何能力結論或訓練效果結論必須先完成四層隔離**：system replay、data manifest diff、eval harness diff、model checkpoint diff；若某一層與事故無關，應標記為 `N/A` 並寫明理由。**證據不齊時，只能登記為未歸因事故，不能寫入能力結論。**

### 7.1 事故分級：觀察、警報、阻斷

| 級別 | 觸發訊號 | 現場動作 | 可否繼續訓練 |
| --- | --- | --- | --- |
| 觀察 | 單個 eval 小幅波動、分桶 loss 輕微偏移、單次 seed 接近閾值 | 記錄 incident_id，復跑相鄰 checkpoint 或固定樣本籃子 | 可以繼續，但不能 promotion |
| 警報 | 連續 checkpoint 指標異常、public/private 分歧、某資料桶 loss 明顯偏離、目標能力漲但 guardrail 變壞 | 凍結變數，拉 Data/Eval/Training DRI 做 30 分鐘首查 | 視 guardrail 決定；通常暫停放大 |
| 阻斷 | NaN/Inf、resume 不連續、汙染或 license 不清、不可犧牲能力越線、離線可用但線上不可服務 | 停止 run 或停止釋出，回滾到最近健康 checkpoint，開事故覆盤 | 不繼續，除非覆盤後新建 experiment_id |

處理原則：

| 原則 | 具體做法 |
| --- | --- |
| 先凍結現場 | 儲存 checkpoint、optimizer state、訓練日誌、data manifest、eval raw outputs、系統指標和相關 commit |
| 先分層歸因 | 分清 system、data、eval、model、post-train recipe 哪一層先變了 |
| 先看分桶 | 全域性 loss 正常不代表資料桶、語言、長度、任務都正常 |
| 先保護 guardrail | 不可犧牲能力超閾值回撤時，不用等目標能力繼續上漲 |
| 不做隱性修補 | 改過濾器、取樣權重、prompt、harness、judge、decoding 或 serving config 都必須新建 experiment_id |

### 7.2 訓練中事故 triage 表

| 事故型別 | 現場症狀 | 30 分鐘首查 | 分層歸因 | 立即決策 | 必須留檔 |
| --- | --- | --- | --- | --- | --- |
| numerical | NaN / Inf / loss 爆炸 | 梯度範數、overflow 計數、LR/warmup、最近 batch、optimizer state、resume 日誌 | system：精度/kernel；data：壞樣本；model：LR/clip；checkpoint：resume | 阻斷；回滾到最近健康 checkpoint，replay 問題 batch | `checkpoint_id`、batch id、overflow log、optimizer hash、replay 結果 |
| numerical / model | 階段切換後 loss spike | 階段切換 commit、LR schedule、mix 權重、長度分佈、optimizer/RNG 是否連續 | data：mix 突變；model：schedule；system：resume；eval：視窗口徑 | spike 可解釋且快速恢復則觀察；持續異常則 rollback | phase-change diff、前後 3 個 checkpoint 曲線、data manifest diff |
| infra | 吞吐下降 / GPU 利用率異常 | tokens/s、MFU、dataloader wait、NCCL、儲存 IO、kernel 版本、batch shape | system 優先；確認前不要解釋為模型變化 | 暫停能力結論；必要時停 run 排查 | system metrics、profile trace、程式碼/映象版本、復跑對比 |
| data | 全域性 loss 正常但某桶 loss 異常 | bucket validation loss、sampling weight、tokenizer/packing、manifest diff、樣例抽查 | data 優先；再看 tokenizer/packing 和 eval bucket | 凍結或下線問題桶；嚴重時 rollback manifest | bucket id、manifest hash、抽樣樣例、過濾器版本、修復 diff |
| data / capability | 合成資料模板化或錯誤率升高 | teacher/version、prompt、verifier、template n-gram、response length、人工錯誤率 | data：生成器；eval：樣例太窄；model：學到格式而非能力 | 降低 synthetic ratio 或隔離桶；不把格式收益寫成推理收益 | generator config、verifier 版本、重複率、人工審查記錄 |
| capability | 通用能力早期大幅回撤 | base regression max drawdown、低分桶、語言/領域固定籃子、資料 mix | model：LR/訓練過長；data：專項比例過高；eval：抽樣偏差 | 觸發 guardrail；回滾或降低專項比例 | scorecard、失敗樣例簇、mix curve、rollback 點 |
| capability | 目標能力上漲但 guardrail 變壞 | 目標指標、不可犧牲能力、相鄰 checkpoint、錯誤型別變化 | model/data 權衡；確認不是 eval 口徑變化 | 不 promotion；只有 trade-off 被業務接受且可復現才 extend | Pareto 曲線、guardrail 閾值、DRI 簽字、樣例對照 |
| eval | Eval 突然大漲/大跌 | data_hash、harness_commit、prompt/wrapper、judge、decoding、答案抽取器 | eval 優先；用同一 checkpoint 跑新舊 harness diff | 凍結能力結論；舊/新 harness 交叉復跑 | eval diff、raw outputs、judge 版本、metric 變更說明 |
| eval / data | Public benchmark 漲但 private 不漲 | contamination report、private holdout、題型相似度、raw output | data 汙染或 eval 過擬合；也可能 public 太窄 | 從 primary 指標降權；不能放大 | contamination 證據、holdout 使用記錄、剔除樣本列表 |
| capability / context | Needle 好但真實長任務差 | 長任務 raw output、證據位置、資訊密度、不可回答樣本、長度分佈 | eval：needle 過窄；data：缺綜合/衝突任務；system：serving 差異 | 不把 needle 作為主 gate；暫停長訓放大 | 長任務失敗簇、證據定位、長度桶分數、serving 配置 |
| capability / context | 長上下文訓練傷短任務 | sequence_length_distribution、短任務迴歸、packing/padding、短樣本佔比 | data：長樣本稀釋；model：位置/attention 適應副作用 | 回滾長度 mix 或恢復短樣本比例 | 長度分佈 diff、短任務 scorecard、curriculum 版本 |
| infra / context | 離線長上下文可用但線上不可用 | serving config、KV cache、chunked prefill、batching、kernel、latency/cost | system 優先；模型能力結論暫緩 | 系統阻斷；不釋出，不把線上失敗記成模型失敗 | offline/online config diff、線上 raw output、profile |
| post-train handoff | Post-train 後收益消失 | paired protocol、seed、SFT/RL 資料版本、checkpoint selection、訓練曲線 AUC | post-train recipe 或 base eval 假陽性；也可能 mid-train 不穩 | 不放大；重複 paired run 或回到目標資料 | recipe hash、seed、A/B 曲線、post-train raw outputs |
| post-train handoff | Reward hacking 增加 | RL raw output、reward/verifier、拒答/安全、格式依賴、CoT 模板 | post-train reward 漏洞；mid-train 可能提前固化格式 | 降低行為格式資料；修 reward/verifier 後重跑 | reward 版本、hacking 樣例、行為資料比例、修復記錄 |
| infra | Checkpoint resume 不連續 | resume 前後 loss、optimizer/RNG、data cursor、checkpoint hash、訓練程式碼版本 | system/checkpoint 優先；排除後再看 data/model | rollback；replay resume 測試透過後新建 run | checkpoint lineage、RNG/data cursor、replay log、程式碼 hash |
| eval | Checkpoint 間結論反覆 | 相鄰 checkpoint、CI/方差、多 seed、selection policy、raw outputs | eval 噪聲或 selection 偏差；model 真實波動需二次驗證 | 不用單點決策；報告曲線而非最好點 | selection policy、完整曲線、置信區間、失敗樣例 |

### 7.3 資料事故 runbook

資料事故的第一原則是隔離版本，不是現場修補。任何過濾器、生成器、去重、取樣權重、tokenizer 或 packing 變化，都必須產生新的 data_version 和 manifest hash。

| 步驟 | 動作 | 退出條件 |
| --- | --- | --- |
| 1. 凍結 | 儲存舊 manifest、取樣權重、生成器配置、過濾器版本和抽樣樣例 | 能復現事故前的資料輸入 |
| 2. Diff | 對比 token 數、語言/領域/長度分佈、重複率、汙染率、壞樣本率 | 解釋至少一個異常分桶 |
| 3. 隔離 | 下線或降權問題桶，保留通用保真桶和 guardrail eval | 問題桶不再進入繼續訓練 |
| 4. 復跑 | 用固定 checkpoint 復跑 data validation 和 linked eval | 證明修復來自資料，而不是 eval 改口徑 |
| 5. 覆盤 | 更新資料桶准入卡和 prevention_rule | 新 data_version 才能進入下一次訓練 |

如果汙染、license 或隱私邊界無法解釋清楚，應直接 kill 該資料版本。對 mid-train 來說，“可能汙染但分數好”不是可接受狀態。

### 7.4 訓練事故 runbook

訓練事故優先保護可恢復性。發現 numerical 或 resume 類問題後，不要先調學習率碰運氣；先確認 checkpoint、optimizer、RNG、data cursor 和程式碼版本是否能 replay。

| 步驟 | 動作 | 退出條件 |
| --- | --- | --- |
| 1. 停表 | 標記 incident_id，凍結當前 run，不再寫入能力結論 | 現場日誌和狀態完整 |
| 2. 回放 | 從最近健康 checkpoint 回放短視窗，確認是否可復現 | 能復現則定位，不能復現則記錄系統不確定性 |
| 3. 單變數修復 | 每次只改一個變數：LR、clip、precision、bad batch、resume 邏輯之一 | 修復與指標變化可歸因 |
| 4. 迴歸 | 重新跑分桶 loss、base regression 和關鍵 target eval | 沒有新增 guardrail 事故 |
| 5. 決策 | kill / retry / rollback / extend / promote | 有 DRI 簽字和證據路徑 |

訓練繼續的最低條件是：數值穩定、系統 replay 成功、資料 manifest 未漂移、eval harness 未漂移。缺任一項，只能 retry 或 rollback，不能 promote。

### 7.5 評估事故 runbook

評估事故最常見的錯誤，是把工具變化當成能力變化。只要 prompt、wrapper、judge、答案抽取器、decoding、樣本集或 metric 任一項變化，舊分數和新分數都不能直接橫比。

| 排查矩陣 | 舊 checkpoint | 新 checkpoint |
| --- | --- | --- |
| 舊 harness | 復現歷史基線 | 看模型真實增量 |
| 新 harness | 看工具口徑變化 | 只作為新口徑結果 |

只有當“舊 harness 下新 checkpoint 變好”和“新 harness 下舊 checkpoint 沒有同等變好”同時成立，才可以把收益初步歸因給模型。若 private holdout 被反覆檢視或調參，應降級為 dev-private，並重新建立 sealed-private 或 final-blind。

### 7.6 Post-train 接力事故

Mid-train 的收益如果在 post-train 後消失，先不要簡單判定 mid-train 無效。需要同時排查四件事：

| 排查項 | 可能問題 | 處理 |
| --- | --- | --- |
| recipe 是否凍結 | SFT/RL 資料、seed、預算、reward/verifier、decoding 變化 | 重跑 paired protocol |
| selection 是否偏差 | 只挑了 post-train 最好點或 base 最好點 | 用固定 checkpoint selection policy |
| 能力是否被覆蓋 | SFT 風格資料壓過了目標能力，RL 獎勵偏向短答或格式 | 若調整 post-train 資料/獎勵，必須新建 experiment_id 或 paired_protocol_version 後再測 readiness |
| 收益是否是假陽性 | base eval 汙染、public 過擬合、樣例太窄 | 回到第 6 章 eval registry 和第 7.5 節 |

**如果 post-train 接力連續兩輪在 paired protocol 下失敗，且 base 端收益只存在於 public benchmark 或單個 seed，應 kill 該 mid-train 方向**；如果 base 端收益穩定但被 SFT/RL 覆蓋，則把結論寫成“post-train handoff 未打通”，不要寫成“mid-train 無收益”。為排查覆蓋問題而調整 SFT/RL 資料、reward、verifier、預算或 seed 時，舊 paired protocol 的結論不能繼續作為 mid-train 收益證據，只能作為診斷線索。

### 7.7 事故覆盤模板

每個警報或阻斷級事故都要有一條覆盤記錄。覆盤不是追責文件，而是讓下一次訓練能少踩同一個坑。

```text
incident_id:
experiment_id:
incident_type: numerical | data | eval | infra | capability | post-train handoff
severity: observation | alert | blocker
first_detected_at:
affected_checkpoint:
symptom:
root_cause: confirmed | suspected | unresolved
changed_variables:
rollback_point:
evidence_paths:
four_layer_isolation:
  system_replay:
  data_manifest_diff:
  eval_harness_diff:
  model_checkpoint_diff:
decision: kill | retry | rollback | extend | promote | unresolved
prevention_rule:
owner:
reviewed_by:
```

如果 `root_cause` 仍是 `unresolved`，事故可以關閉訓練動作，但不能關閉認知結論：文件中只能寫“未歸因事故，已隔離”，不能寫“模型能力退化”或“某資料桶有效”。

## 8. 給新團隊的 90 天路線圖

這張路線圖是參考節奏，不是日曆承諾。**進入下一階段應由 Gate 0-5 決定，而不是由第幾天決定。**

### 8.0 Gate dashboard：每一階段只回答一個決策問題

90 天路線圖的用途不是排期，而是把專案從“想訓練”推進到“證據足夠再訓練”。每個 gate 只回答一個問題：

| Gate | 決策問題 | 必備證據 | Blocker | Owner | Next action |
| --- | --- | --- | --- | --- | --- |
| Gate 0 | 這個目標是否真的需要 mid-train | charter、不可犧牲能力、v1 eval registry、v1 post-train paired protocol | 目標可由 SFT/RAG/產品邏輯解決；eval 或 holdout 未凍結 | Model Lead | go / no-go charter |
| Gate 1 | 資料是否可以進入實驗 | data manifest/hash、license、汙染檢查、抽樣審查、linked eval | 資料來源不清、合成生成器讀過 eval、holdout 洩漏 | Data DRI | data readiness memo |
| Gate 2 | proxy ablation 是否排除了壞方向 | 目標收益、通用回撤、兩個 scale 或 target 短程訊號、失敗樣例簇 | 只有 public benchmark 漲；合成資料模板化；proxy 結論不穩定 | Eval DRI | ablation decision memo |
| Gate 3 | 主模型短程是否值得繼續 | checkpoint 曲線、分桶 loss、scorecard、事故記錄、Pareto 結果 | NaN/loss spike、resume 不連續、guardrail 越線、長上下文傷短任務 | Training Infra DRI | extend / rollback / kill |
| Gate 4 | 收益能否被 post-train 保留 | paired SFT/RL 結果、seed、AUC、raw outputs、reward/safety 迴歸 | base 漲但 post-train 後消失；recipe 中途改變；reward hacking 增加 | Post-train DRI | readiness report |
| Gate 5 | 是否進入放大訓練或收縮重做 | go/no-go memo、model registry、rollback checkpoint、成本收益判斷 | 收益來自汙染/窄評估/單 seed；事故未歸因；無法解釋回撤 | Decision owner | scale / shrink / stop |

> [!CAUTION]
> **只要某個 gate 的證據不完整，就不要用”再訓一點看看”繞過去。** Mid-train 的專案成本高，**真正的節奏控制來自 gate，而不是日曆**。

每個 gate 的 dashboard 記錄還必須包含 `status`、`owner`、`evidence_path`、`version_hash`、`stale_after` 和 `next_action`。**沒有證據路徑的綠色狀態，應視為黃色。**

### 第 0-14 天：定義目標和評估

對應 Gate 0，主責為 Model Lead、Eval DRI 和 Post-train DRI；本階段的決策會議只回答一個問題：這個目標是否真的需要 mid-train。

交付物：

- Mid-train charter：目標能力、不可犧牲能力、目標 checkpoint、預算。
- Eval matrix：base eval、能力 eval、長上下文 eval、post-train readiness。
- 資料桶設計文件。
- 風險清單和 go/no-go gate。
- Frozen post-train recipe：SFT/RL 資料版本、訓練預算、seed、harness、評估集和失敗判定。

Go/no-go：

- 如果 eval registry、私有 holdout 和 post-train 接力 recipe 沒有凍結，不進入訓練。
- 如果私有 holdout 建立時間晚於訓練資料混合，或允許針對私有 holdout 調參，不進入訓練。

### 第 15-35 天：資料和小模型 ablation

對應 Gate 1-2，主責為 Data DRI 和 Eval DRI；所有實驗必須登記到 4.6 的實驗台賬。

交付物：

- 例如 1B/3B/7B proxy model 上的資料 mix 實驗；具體 scale 由團隊現有模型譜系和預算決定。
- 學習率和 token budget sweep。
- 合成資料正確性和多樣性報告。
- 每個資料桶的 loss 和能力關聯分析。
- Data manifest/hash、near-duplicate 檢測報告、汙染檢查報告、人工抽檢樣本。

Go/no-go：

- 至少一個 mix 在目標能力上提升，且通用能力回撤可接受。
- 合成資料沒有明顯汙染和風格坍縮。
- Proxy 成功只說明壞方向被排除，不能證明 target scale 一定成立；進入主模型短程前，至少需要兩個 scale 方向一致，或 target model 短程 run 驗證關鍵訊號。

### 第 36-60 天：主模型短程 mid-train

對應 Gate 3，主責為 Training Infra DRI、Eval DRI 和 Model Lead；本階段必須先定義 kill switch，再啟動訓練。

交付物：

- 主模型短程訓練，例如 10B-100B token 級視窗；具體 token budget 由 Gate 2 證據和訓練成本決定。
- 每日 checkpoint eval。
- 能力/遺忘 Pareto 曲線。
- 候選 checkpoint 選擇。
- 訓練 runbook、checkpoint lineage、resume 記錄、訓練程式碼 commit、事故記錄。

Go/no-go：

- 目標能力提升不是單一 benchmark 驅動。
- 通用能力下降低於事先定義閾值。
- 生成樣例沒有明顯風格漂移。
- 出現 NaN、不可恢復 loss spike、梯度異常、吞吐異常、某資料桶 loss 反常、checkpoint resume 不連續，或長上下文訓練傷害短任務時，觸發停止或回滾。

### 第 61-75 天：post-train 接力驗證

對應 Gate 4，主責為 Post-train DRI 和 Eval DRI；如果 post-train recipe 在本階段改變，mid-train 結論應視為無效或重新標記實驗 ID。

交付物：

- 固定 SFT recipe 對比。
- 如適用，固定 RL/RLVR recipe 對比。
- post-train 後真實任務評估。
- reward hacking / safety / style 迴歸報告。
- Post-train readiness report：recipe 版本、seed、預算、harness、失敗判定和收益保留比例。

Go/no-go：

- Mid-trained checkpoint 在 post-train 後仍保留收益。
- 對齊難度沒有顯著增加。

### 第 76-90 天：放大或收縮決策

對應 Gate 5，主責為 Decision owner、Model Lead 和 Release owner；本階段不是自動放大，而是決定繼續、收縮、重做資料或停止。

交付物：

- 放大候選 recipe 或下一階段收縮重做 recipe。
- 是否進入 100B-1T+ token 訓練的決策。
- 資料和評估自動化改進清單。
- 長上下文或第二能力目標的下一階段計劃。
- Model registry：checkpoint lineage、data manifest、eval raw outputs、訓練日誌、許可權、保留週期和可回滾 checkpoint。

Go/no-go：

- **如果收益主要來自汙染、窄評估或 post-train 偶然性，不放大。**
- 如果收益穩定、可解釋、可在 post-train 後保留，進入主訓練。

## 9. 可執行 checklist

### 9.0 最小交付包與交付物索引

如果團隊不想一開始就維護完整文件體系，至少要有下面 8 個交付物。缺任意一個，mid-train 結論都很難覆盤。

| 交付物 | Owner | 模板位置 | 凍結 Gate | 用在哪個會議 |
| --- | --- | --- | --- | --- |
| Mid-train charter | Model Lead | 0.2 / 8.0 | Gate 0 | 啟動會、go/no-go 會 |
| Data manifest | Data DRI | 4.2.1 / 9.1 | Gate 1 | 資料 readiness 會、事故覆盤 |
| Eval registry | Eval DRI | 6.1 / 9.3 | Gate 0-1 | eval freeze 會、checkpoint review |
| Training runbook | Training Infra DRI | 4.3.1 / 5 / 7.4 | Gate 3 | train launch 會、事故處理 |
| Experiment ledger | Model Lead / PM | 4.6 | Gate 0 後持續更新 | 每週實驗例會 |
| Checkpoint scorecard | Eval DRI | 6.7 | Gate 3-5 | checkpoint review、放大決策 |
| Post-train readiness report | Post-train DRI | 6.6 / 7.6 | Gate 4 | post-train handoff 會 |
| Incident review | 事故 owner | 7.7 | 警報/阻斷事故後 | 事故覆盤、prevention review |
| Go/no-go memo | Decision owner | 8.0 / 9.4 | Gate 5 | 放大、收縮或停止決策 |

Checklist 的作用是驗收這些交付物，不是製造流程負擔。對新團隊來說，先把最小交付包跑通，比追求完美平台更重要。

每個 checklist 項都不應只打勾，還要繫結最小審計欄位：

```text
owner:
evidence_path:
version_hash:
stale_after:
gate_role: primary | guardrail | diagnostic
status: green | yellow | red
```

如果一項檢查沒有 `evidence_path` 或 `version_hash`，它最多隻能算 yellow，不能作為 gate 透過證據。

### 9.1 資料 checklist

- [ ] 每個資料版本都有 data manifest/hash，記錄 source snapshot、processing commit、tokenizer、dedup、contamination、sampling weight、shuffle seed。
- [ ] 每個資料來源有 license / 合規記錄。
- [ ] 每個資料桶有質量打分和人工樣例審查。
- [ ] 每個資料桶繫結 linked_eval_id，未繫結 eval 的桶不提高取樣權重。
- [ ] benchmark contamination 檢查完成。
- [ ] 訓練集、驗證集、私有 holdout 分離。
- [ ] 合成資料記錄 teacher/version、prompt、取樣引數、verifier、過濾規則、拒樣比例、模板重複率、人工錯誤率。
- [ ] 合成資料有正確性過濾、多樣性、模板化和 style entropy 檢查。
- [ ] 長上下文資料包含真實長任務，不只是拼接文字。
- [ ] 通用資料保留比例有 ablation 支援。

### 9.2 訓練 checklist

- [ ] Base checkpoint eval 完成。
- [ ] Tokenizer / special tokens 凍結或變更有充分理由。
- [ ] Optimizer state 接續策略明確。
- [ ] LR schedule、warmup、peak/min LR、decay shape、gradient clipping、precision 記錄清楚。
- [ ] global batch、microbatch、packing/masking、data order、shuffle seed 記錄清楚。
- [ ] checkpoint selection policy 在訓練前凍結。
- [ ] 每個階段有 checkpoint 儲存策略。
- [ ] 分桶 validation loss 可觀測。
- [ ] MoE/attention/long-context 系統指標可觀測。
- [ ] 訓練可 resume，checkpoint 可復現。

### 9.3 評估 checklist

- [ ] Eval registry 已凍結，包含 eval_id、eval_family、data_hash、harness_commit、prompt_or_wrapper_version、metric、decision_role、threshold_or_guardrail、raw_output_path。
- [ ] Base eval、能力 eval、長上下文 eval 分開。
- [ ] 每個 eval 標明 primary、guardrail 或 diagnostic，primary 指標不超過 3-5 個。
- [ ] 指標包含真實任務和人工審查。
- [ ] 不只看平均分，也看能力回撤。
- [ ] 每個候選 checkpoint 生成 scorecard，包含目標能力 delta、最大回撤、汙染狀態、失敗樣例和建議動作。
- [ ] 每個候選 checkpoint 做同一套 eval。
- [ ] 私有 holdout 分為 dev-private、sealed-private、final-blind，建立時間早於訓練資料 mix 凍結，且不得針對 sealed/final 調參。
- [ ] 長上下文 eval 覆蓋長度、位置、資訊密度、任務型別和部署一致性。
- [ ] Post-train 接力 recipe 固定，並記錄資料順序、seed、harness、預算、checkpoint selection policy 和失敗判定。
- [ ] Post-train 後收益可復現。

### 9.4 組織 checklist

- [ ] 資料團隊、訓練團隊、Post-train DRI 共用同一張實驗台賬。
- [ ] 每次實驗記錄資料 mix、LR、token budget、context length、eval 版本。
- [ ] Research、Data、Training、Eval、Post-train、Release owner 均已明確。
- [ ] Model Lead、Data DRI、Training Infra DRI、Eval DRI、Post-train DRI、Safety/Compliance DRI、Product/Domain DRI、Decision owner 均已明確。
- [ ] Mid-train charter、data manifest/hash、eval registry、runbook、checkpoint ledger、post-train readiness report、go/no-go memo 均有 owner。
- [ ] 專案開始前凍結“不可犧牲能力”和各階段回滾條件。
- [ ] 每個 gate 的閾值、阻斷條件和簽字人已在實驗前寫入 gate memo。
- [ ] 任何中途改 mix、改過濾器、改 eval 版本的行為都會新建 experiment_id。
- [ ] 訓練中有停止條件。
- [ ] 放大訓練前必須過 go/no-go gate。
- [ ] 釋出前保留 pre-mid、mid、post 三類 checkpoint 的完整報告和 model registry。

## 10. 對有 post-train 經驗的基模團隊的特別提醒

第一，**不要用 SFT 的心智做 mid-train**（詳見 1.2、1.4）。Mid-train 資料可以包含 QA、CoT、程式碼任務，但其目標不是讓模型學會產品回答格式，而是改變能力分佈。**自檢：如果資料 mix 大部分是指令-回答對，大機率在做提前的 SFT，而不是能力塑形。**

第二，**不要讓 RL 背所有鍋**（詳見 2.4 第四點、6.6）。如果 base model 沒有足夠的數學、程式碼、長上下文和自我糾錯 substrate，RL 很可能只學到表面搜尋或冗長推理。**自檢：如果目標任務的 zero-shot pass@100 為零，說明能力不在 base 的可達邊界內，RL 大機率無法穩定探索到正確解。**

第三，**不要只追榜**（詳見 6.4、7.2 eval 事故、8.0）。Mid-train 的真實收益要在私有任務、真實產品任務和 post-train 接力中同時成立。**自檢：如果某個 eval 收益超過 5 個百分點但私有 holdout 不動，先假設是汙染或格式適配。**

第四，**不要一開始就追 1M context**（詳見 4.5）。對多數團隊，先把 32K/128K 做到可靠，比展示極長 context 更實際。**自檢：如果短任務（<4K）效能在拉長後明顯下降，說明 context extension 的成本還沒控制住。**

第五，**不要忽視資料工程**（詳見 4.2、4.2.1、9.1）。Mid-train 專案中，訓練程式碼常常不是瓶頸，資料 lineage、汙染控制、分桶 eval 和實驗紀律才是。**自檢：如果團隊無法在 30 秒內說出當前訓練使用的 data manifest hash 和汙染檢查版本，資料治理還沒到位。**

## 11. 第一輪搜尋空間：示例搜尋網格，不可直接作為預設 mix

這一章是工作樣例，不是配方。它的用途是幫助第一次做 mid-train 的團隊把資料、訓練、評估和 post-train 接力串成一個最小證據鏈（”最小閉環”的定義見 4.0）。

如果團隊已有一個可用 base model 和穩定 post-train pipeline，建議第一輪不要做大而全，先做一個”能力型 mid-train + post-train 接力”的最小閉環。以下比例不是推薦 recipe，而是用於啟動 ablation 的搜尋空間；不同模型規模、預訓練歷史、目標語言和產品任務會顯著改變最優 mix，主訓練前必須透過 proxy scale 和短程主模型驗證。下列比例和 token 區間只用於設計 ablation 變數，不得作為預設 mix、預設 LR 或預設預算。

只有在 base checkpoint、目標能力、不可犧牲能力、eval registry、data manifest 和 post-train paired protocol 已凍結後，下面的區間才可作為 ablation 變數；否則不應進入訓練配置。

目標：

- 提高數學、程式碼、STEM 和複雜 QA。
- 不顯著傷害通用、多語、事實和自然生成。
- 驗證 SFT/RL 後是否仍有收益。

資料 mix 搜尋空間（bucket sensitivity sweep 軸）：

- 高質量通用文字：用低/中/高几個點測試保真底線，例如 30/50/70%，不是固定佔比建議。
- 程式碼與 FIM：用低/中/高几個點測試程式碼能力和普通生成漂移，例如 10/20/30%。
- 數學/STEM：用低/中/高几個點測試推理注入收益、飽和點和通用回撤。
- 合成教材/可驗證推理：用上限 sweep 測試 verifier 質量、模板化和 teacher 錯誤放大。
- QA/CoT/短答任務形態：用單獨開關或低比例混入測試格式漂移，避免把 post-train 行為格式提前固化。

這些 bucket 軸只用於第一輪單變數或少變數搜尋，不是要同時滿足的歸一化比例表。實際 mix 必須透過小模型和短程主模型實驗決定。

這個起點不是任何公開模型的復刻配方，也不是從公開報告反推出的最優比例。它只是一組便於新團隊暴露資料衝突、遺忘風險和合成資料邊界的搜尋網格。

主訓練前必須有 mix sensitivity curve，而不是隻選擇一個看起來最好的點。至少回答：

- 通用資料比例下降時，哪些不可犧牲能力最先回撤？
- 數學/STEM 或程式碼比例提高時，收益是否線性、飽和還是反轉？
- Synthetic ratio 增加時，私有任務、風格審查和模板重複率是否同步惡化？
- 同一 mix 在不同 seed 或相鄰 checkpoint 上方向是否一致？
- 目標能力收益是否足以補償通用回撤和訓練成本？

第一輪不要只跑一個 mix。更好的做法是保留一個 control mix，再圍繞單一變數做小矩陣：

| 變體 | 改什麼 | 不改什麼 | 目的 |
| --- | --- | --- | --- |
| Control | 低風險通用保真 mix | LR、token budget、context length | 建立迴歸底線 |
| Math/STEM up | 只提高數學/STEM | 程式碼、通用、LR 不變 | 判斷推理注入收益和遺忘代價 |
| Code/FIM up | 只提高程式碼/FIM | 數學、通用、LR 不變 | 判斷程式碼能力和普通生成漂移 |
| Synthetic cap | 限制 synthetic 上限 | 其它桶不變 | 判斷合成資料是否帶來模板化 |
| General retain | 提高高質量通用資料 | 目標桶不變 | 判斷通用保真能否降低迴撤 |
| LR lower | 只降低 LR | 資料 mix 不變 | 區分資料問題和最佳化問題 |

每個變體只能改 1-2 個主變數。邊訓邊改 mix、過濾器、LR 或 eval 版本，會讓實驗無法解釋。

透過 Gate 2 後的主模型短程驗證起點：

- 低學習率繼續訓練。
- 先用原 context length 或小幅擴充套件。
- 10B-50B tokens 級短程主模型驗證上限示例；實際預算必須由 proxy 結果、target model 短程訊號、遺忘閾值和成本共同決定。
- 每 10%-20% token 儲存 checkpoint 並 eval。

評估起點：

- Base eval：MMLU/MMLU-Pro、BBH、GPQA、GSM8K/MATH、HumanEval/MBPP、多語集。
- 私有 eval：內部數學、程式碼、複雜 QA、真實使用者任務。
- Post-train readiness：固定 SFT recipe，對比 pre-mid 和 mid checkpoint。

如果這輪閉環沒有穩定收益，不要急著擴大 token budget。先回到資料和學習率。

### 12.4 推薦閱讀順序

1. 先讀目標模型家族最新報告；沒有目標家族時，先讀 Qwen3 和 DeepSeek-V3。
2. 只在做 1M/超長上下文時補讀 Qwen2.5-1M。
3. 只在需要明確 mid-training 治理正規化時補讀 OLMo 2。
4. 只在合成資料是主變數時補讀 Phi-4。
5. 只在需要 late-stage/annealing 公開細節時補讀 Llama 3。
6. 最後讀 survey、PRISM 和 pre/mid/RL 論文，把機制假設轉成自己的 ablation。
