# 投資論文漂移檢測：分清事實變化與措辭變化

對 $ARGUMENTS 執行投資論文漂移檢測。

**支援輸入格式**：
- `公司名 舊報告路徑 新報告路徑` — 指定兩份研究報告或論文快照進行對比
- `公司名 reports/{公司名}-thesis-舊日期.md reports/{公司名}-thesis-新日期.md` — 對比兩份帶日期的論文快照
- `公司名` — 自動查詢 `reports/{公司名}-thesis.md` 及同目錄歷史快照；如果沒有基線則轉入缺失基線處理

> "當事實改變時，我就改變想法。你呢？" —— 凱恩斯
>
> "股價波動不是論文漂移，事實變了才是。" —— AI Berkshire

## 設計理念

長期持倉最難的不是每天讀新聞，而是區分三件事：
- **事實改變**：收入、利潤率、競爭格局、管理層行為、資本配置發生可驗證變化
- **價格改變**：市場情緒或估值倍數變化，但生意本身未變
- **措辭改變**：兩份報告表達不同，但底層證據和判斷沒有變化

投資論文漂移檢測的目標是：**只在證據變化時承認論文變化**。不能因為報告換了寫法就製造漂移，也不能因為股價漲跌就誤判基本面。

本 Skill 依賴 `/thesis-tracker` 輸出的結構化維度：核心假設清單、紅線清單、估值錨點、追蹤記錄表。沒有這些結構時，先補齊基線，再做漂移檢測。

## 執行流程

### 第一步：判斷操作模式

解析 `$ARGUMENTS`：
- 如果提供兩份報告路徑 → 進入**指定報告對比**模式
- 如果只提供公司名 → 查詢 `reports/{公司名}-thesis.md` 及歷史快照，進入**自動快照對比**模式
- 如果只找到一份報告或沒有歷史基線 → 進入**缺失基線處理**模式
- 如果兩份報告不是同一家公司 → 停止並要求使用者確認，不做跨公司漂移判斷

## 模式A：指定報告對比

### A1：讀取並校驗兩份報告

讀取舊報告和新報告，提取：
- 報告日期、公司名、股票程式碼
- 核心論文（5句話）
- 核心假設清單
- 紅線清單
- 估值錨點
- 追蹤記錄表
- 管理層質量判斷
- 競爭護城河判斷
- 當前建議動作（買入 / 持有 / 觀察 / 減倉 / 清倉）

如果報告缺少關鍵結構，先標註"結構缺失"，但仍儘量從正文中抽取證據；抽取不到的維度標為"無法判斷"，不能編造結論。

### A2：證據歸一化

把兩份報告中的事實證據整理成同一張表：

| 維度 | 舊報告證據 | 新報告證據 | 資料來源 | 是否可驗證 |
|------|-----------|-----------|---------|-----------|
| 估值錨點 | | | | |
| 核心假設 | | | | |
| 紅線 | | | | |
| 管理層質量 | | | | |
| 競爭護城河 | | | | |

**只比較證據，不比較文風。** 如果新舊報告只是同義改寫、排序變化、語氣變化，但事實資料和判斷閾值沒有變化，判定為 Unchanged。

### A3：數值與估值校驗

所有數值變化必須使用 `tools/financial_rigor.py` 做精確計算，禁止 LLM 心算：

```bash
python3 tools/financial_rigor.py verify-valuation \
  --price {當前價格} \
  --eps {EPS} \
  --bvps {每股淨資產} \
  --fcf-per-share {每股自由現金流}
```

如需計算市值、百分比變化、目標價差異或情景估值，使用：

```bash
python3 tools/financial_rigor.py verify-market-cap --price {價格} --shares {股本} --reported {報告市值} --currency {幣種}
python3 tools/financial_rigor.py cross-validate --field {欄位} --values '{JSON}' --unit {單位}
python3 tools/financial_rigor.py three-scenario --price {價格} --eps {EPS} --shares {股本億} --growth {樂觀} {中性} {悲觀} --pe {樂觀PE} {中性PE} {悲觀PE}
python3 tools/financial_rigor.py calc --expr '{精確算式}'
```

關鍵財務資料必須至少兩處獨立來源交叉驗證。來源不足、口徑不一致、無法複核的數字必須標註為"低置信度 / 待核實"。

### A4：逐維度判定漂移

固定使用以下維度，不要臨時增減：

| 維度 | 判定重點 | Improved | Unchanged | Weakened |
|------|---------|----------|-----------|----------|
| 估值錨點 | 內在價值、PE/PB/FCF Yield、安全邊際、目標價區間 | 安全邊際擴大或內在價值上修且經工具驗算 | 估值區間和安全邊際無實質變化 | 安全邊際收窄、內在價值下修或估值假設失效 |
| 核心假設清單 | 收入增速、利潤率、現金流、使用者/訂單/產能等可驗證假設 | 更多假設被新證據強化 | 假設狀態與證據基本一致 | 假設邊際弱化、受損或破裂 |
| 紅線清單 | 誠信、監管、業務衰退、競爭突破、管理層異常動作 | 原有紅線風險解除或顯著下降 | 未觸發且風險水平不變 | 紅線被觸發或觸發機率上升 |
| 管理層質量 | 誠信、資本配置、回購分紅、執行力、股東友好度 | 新行為提高信任度 | 行為延續舊判斷 | 行為損害信任或資本配置變差 |
| 競爭護城河 | 市佔率、定價權、網路效應、成本優勢、替代威脅 | 護城河變寬或競爭優勢被驗證 | 格局無實質變化 | 護城河被削弱或競對突破 |

每個維度只能給出三類結論：**Improved / Unchanged / Weakened**。

### A5：證據驅動規則

每個非 Unchanged 的結論必須引用導致變化的具體新證據：
- 財報行專案：例如收入增速、毛利率、經營現金流、回購金額、淨現金
- 監管披露：例如 10-K/20-F、年報、中報、港交所公告、SEC filing
- 新聞事件：例如管理層變動、監管處罰、重大客戶流失、競品突破
- 價格與估值：必須說明這是"估值變化"還是"基本面變化"，不能混淆

如果找不到能解釋變化的證據，必須判定為 **Unchanged** 或 **無法判斷**，不能用措辭差異推斷漂移。

### A6：輸出漂移報告

#### 報告結構

```
一、對比物件與時間跨度
二、總體結論：論文是否漂移
三、維度漂移表
四、證據差異明細
五、估值與數值驗算
六、建議動作遷移
七、不確定項與需補充來源
八、下次跟蹤重點
```

#### 維度漂移表

| 維度 | 舊判斷 | 新判斷 | 漂移方向 | 觸發證據 | 置信度 |
|------|-------|-------|:--------:|---------|:------:|
| 估值錨點 | | | Improved / Unchanged / Weakened | | 高/中/低 |
| 核心假設清單 | | | Improved / Unchanged / Weakened | | 高/中/低 |
| 紅線清單 | | | Improved / Unchanged / Weakened | | 高/中/低 |
| 管理層質量 | | | Improved / Unchanged / Weakened | | 高/中/低 |
| 競爭護城河 | | | Improved / Unchanged / Weakened | | 高/中/低 |

**Unchanged 行的觸發證據寫 `—`，不要為了填表編造證據。**

#### 總體結論必須回答

1. **論文是否漂移？** 未漂移 / 正向漂移 / 負向漂移 / 證據不足無法判斷
2. **漂移來自哪裡？** 估值 / 基本面 / 管理層 / 競爭格局 / 紅線事件
3. **是事實變化還是價格變化？** 明確拆開說明
4. **建議動作如何遷移？** 例如：Watch → Buy、Buy → Hold、Hold → Reduce、Reduce → Exit
5. **下一步需要什麼證據？** 下一份財報 / 監管披露 / 管理層說明 / 競對資料

## 模式B：自動快照對比

### B1：查詢快照

在 `reports/` 中查詢：
- `reports/{公司名}-thesis.md`
- `reports/{公司名}-thesis-*.md`
- `reports/{公司名}/` 目錄下包含 `thesis`、`論文`、`追蹤` 的報告

選擇時間最早且結構完整的檔案作為舊報告，時間最新的檔案作為新報告。若使用者指定日期，以使用者指定為準。

### B2：防止錯誤配對

對比前必須確認：
- 公司名或股票程式碼一致
- 報告日期不同
- 兩份報告都包含可抽取的論文結構或研究結論

如果無法確認同一公司，停止並要求使用者提供明確路徑。

### B3：執行模式A

找到兩份有效快照後，按模式A完整執行。

## 模式C：缺失基線處理

如果只找到一份報告或沒有找到舊快照：

1. 明確說明：**缺少可比較的歷史基線，不能執行漂移檢測**
2. 不要根據記憶或市場印象補造舊論文
3. 引導使用者先使用 `/thesis-tracker {公司名} 建立論文` 建立結構化基線
4. 如果當前報告已足夠完整，可建議將它儲存為 `reports/{公司名}-thesis.md` 作為未來漂移檢測基線

輸出格式：

```
無法執行論文漂移檢測：缺少歷史基線。

已找到：
- 當前報告：{路徑 / 未找到}
- 歷史基線：未找到

建議：
1. 先執行 /thesis-tracker {公司名} 建立論文
2. 下次有新財報或重大事件後，再執行 /thesis-drift {公司名} 舊報告 新報告
```

## 關鍵原則

- **證據優先於措辭** — 同義改寫不是漂移，只有事實證據變化才是漂移
- **基本面優先於股價** — 股價漲跌隻影響估值錨點，不自動改變生意質量
- **數值必須驗算** — 所有百分比、估值倍數、目標價差異必須用 `tools/financial_rigor.py`
- **不確定就標註不確定** — 來源缺失、口徑不一致、無法複核時，不要硬判
- **紅線單獨處理** — 紅線觸發優先順序高於估值便宜，不能被低 PE 掩蓋
- **輸出必須可覆盤** — 每個 Improved / Weakened 結論都要能追溯到具體證據
